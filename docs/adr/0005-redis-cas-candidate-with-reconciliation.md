# 0005 — Redis CAS как кандидат + reconciliation с Postgres

**Статус:** Accepted

## Контекст

Быстрый reject плохой ставки требует атомарной проверки «лучше текущей И лот открыт» без похода в Postgres на каждую попытку — для этого в Redis живёт Lua CAS ([ADR 0001](0001-ioredis-over-node-redis.md)). Но что если Postgres-транзакция (insert bid + bump `lot.version` + outbox row) упадёт **после** того, как CAS уже принял ставку в Redis? Redis будет думать, что новая ставка лучшая, а в БД её не будет — расхождение двух источников (§6 ТЗ, диаграмма горячего пути в [`docs/diagrams/hot-path-bid.md`](../diagrams/hot-path-bid.md)).

## Решение

Postgres остаётся единственным источником истины; Redis-значение (`lot:{id}:high`) трактуется как **кандидат**, а не факт. На неудаче TX хендлер читает реальную лучшую ставку из БД (`findCurrentBest`) и реконсилит Redis к ней через fenced-обновление ([`cas.service.ts`](../../apps/api/src/platform/redis/cas.service.ts), `reconcileIfCurrent` — заменяет кандидата, только если тот всё ещё равен `bidId`, который откатывается; если конкурентная валидная ставка уже перезаписала кандидата, реконсиляция её не трогает). Вызывается из [`place-bid.handler.ts`](../../apps/api/src/modules/bidding/application/place-bid.handler.ts) в `catch`-ветке горячего пути.

## Последствия

- Redis остаётся быстрым и без блокировок на чтение горячего пути — цена быстрого CAS не в блокировке БД на каждую ставку, а в необходимости явной сверки на failure-путях.
- Fence по `bidId` — обязательное условие: без него реконсиляция после устаревшего сбоя могла бы затереть уже принятую более свежую и валидную ставку конкурента.
- Между CAS-accept и TX-commit есть короткое окно, где Redis временно «врёт» относительно Postgres — приемлемо, потому что окно закрывается либо коммитом (Redis был прав), либо reconciliation (Redis починен) — постоянного рассинхрона не остаётся.
