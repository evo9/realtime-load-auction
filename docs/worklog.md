# Worklog

Журнал выполненных задач: что реально сделано, где это лежит и чем доказано. Запись появляется **сразу после того, как ревьюер вернул PASS, и до отчёта о готовности в чате** — это последнее действие задачи, а не первое действие следующей. Команда `/task-done` — лишь один из способов прогнать закрывающий конвейер, но не триггер записи: без неё запись всё равно обязательна. Записи не редактируются и не удаляются: только append сверху.

Задним числом не дописываем. Если обнаружилась выполненная, но не залогированная задача — не латай журнал молча: скажи об этом и прогони закрывающий конвейер (ревью + зелёные lint/build/test), иначе запись будет декларацией, а не фактом.

Назначение — быстрый ответ на «что уже готово и как это проверялось» в начале новой сессии, без раскопок в git log. Статус задач при этом отмечается галочкой в [tasks/INDEX.md](tasks/INDEX.md); детали — здесь.

Формат записи:

```
## YYYY-MM-DD — <ID задачи> <короткое название>
**Что:** 1–3 строки по существу — что появилось, какое решение принято
**Файлы:** ключевые пути (не полный diff)
**Проверено:** тесты/команды, которые это доказывают
**Заметки:** отклонения от задачи, отложенное, замеченные грабли — опционально
```

Правила:

- Одна запись — одна задача из `docs/tasks/`. Задачи вне INDEX.md логируются как `ad-hoc — <название>`.
- Не логируй непроверенное: нет PASS ревьюера или зелёных lint/build/test — нет записи.
- Никакой статистики изменений, никаких чек-листов тест-плана, никаких упоминаний ассистента (см. [code-style.md](../.claude/rules/code-style.md)).
- Уроки и корректировки — не сюда, а в [lessons.md](lessons.md). Здесь — только факт выполнения.

## 2026-07-17 — M2-04 platform/scheduler: ZSET-планировщик отложенных команд

**Что:** Планировщик отложенного open/close лота поверх Redis ZSET (`auction:schedule:open`/`auction:schedule:close`). Отклонение от буквального скелета §9.7: вместо раздельных `zrangebyscore`→dispatch→`zrem` (три шага с race-окном — два параллельных тика увидели бы один и тот же элемент до его удаления) — `ZSetScheduler.tick()` вызывает Lua-скрипт `SCHEDULER_CLAIM_DUE`, атомарно совмещающий `ZRANGEBYSCORE`+`ZREM` в одной операции; `dispatch()` вызывается уже после атомарного claim, вне Lua. `schedule(setKey, dueAtMs, payload)` — обычный `ZADD`; `payload` обязан быть стабильным идентификатором (лотId) — повторный `schedule` того же payload обновляет score существующего члена (продление/анти-снайп), а не создаёт дубликат — это задокументировано как инвариант в JSDoc. При ошибке `dispatch()` после claim элемент не теряется — `ZADD GT` возвращает его в ZSET с задержкой (`retryDelayMs`); флаг `GT` не откатывает более позднюю дату, если конкурентный `schedule()` успел продлить срок в промежутке. `SchedulerTicker` — `setInterval` в `onModuleInit`/`onModuleDestroy` с `ticking`-флагом реентерабельности (по образцу `OutboxRelay`), на каждом цикле тикает оба ключа (open/close) через опциональный порт `SchedulerDispatchPort` (throwing `NullSchedulerDispatchPort` до будущей проводки auction-модуля — фолбэк создаётся внутри конструктора `SchedulerTicker` через `@Optional()`, не как провайдер модуля, чтобы будущий модуль мог переопределить токен без конфликта). Добавлена секция `AppConfigService.scheduler` (`tickIntervalMs`, `batchSize`, `retryDelayMs`) с `.min()` и `.max()` на всех трёх — учтён урок из M2-03 (там верхнюю границу пропустили на первом проходе).
**Файлы:** `apps/api/src/platform/scheduler/{scheduler.lua,scheduler-dispatch.port,zset-scheduler,scheduler.ticker,scheduler.module}.ts` + `zset-scheduler.integration-spec.ts`; `apps/api/src/config/{env.schema,app-config.service,env.schema.spec}.ts`, `.env.example`.
**Проверено:** `pnpm -C apps/api lint/build/test/test:integration` зелёные. Интеграционные тесты на реальном Redis (testcontainers): (1) due-диспатч — элемент срабатывает после Δ, не раньше; (2) анти-снайп — повторный `schedule` продлевает срок, старый дедлайн мёртв, элемент диспатчится ровно один раз по новому сроку (не дубль); (3) конкурентность — два параллельных `tick()` на 50 due-элементах суммарно дают ровно 50 диспатчей без повторов; (4) retry — сбой dispatch переставляет элемент с `GT`-задержкой, следующий тик доставляет его. Ревью `load-auction-reviewer` — **PASS WITH WARNINGS** на первом проходе (1 WARNING) → **PASS** после фикса. `spec-guardian` — ALIGNED с §5/§8.2/§9.7 (отклонение от буквального скелета признано осознанным улучшением, публичный контракт `schedule`/`tick` не изменён). `pattern-verifier` — все DoD-пункты подтверждены. Security-review — критичных находок нет (Lua-инъекций нет, `unpack` безопасен при `batchSize≤1000`, конфиг с `.min()`/`.max()` без регрессии).
**Заметки:** ревью нашло WARNING — `setInterval` в `onModuleInit` вызывал `tick()` без обработки reject (`void this.tick()`), транзиентная ошибка Redis привела бы к unhandled rejection и (по дефолтному поведению Node) падению процесса — тот же класс проблемы, что был найден и исправлен в M2-02 (`base.consumer.ts`). Исправлено: `.catch()` с логированием через `Logger.error`, `ticking`-флаг сбрасывается независимо в `finally` внутри `tick()`, повторное ревью — PASS. Отложено как non-blocking: нет unit-теста на `SchedulerTicker` (сам примитив `ZSetScheduler` полностью покрыт интеграционными тестами на реальном Redis); DI-паттерн порта (`@Optional()`+фолбэк в конструкторе) отличается от `DEDUP_PORT` в `platform/messaging` (провайдер модуля) — расхождение признано оправданным (throwing-порт не должен быть переопределяемым молча), но зафиксировано как несогласованность стиля между модулями.

## 2026-07-17 — M2-03 platform/outbox: таблица + relay-публикатор

**Что:** Реализован транзакционный outbox, замыкающий уже существующий с M1-03 контракт `OutboxPort.add(manager, eventType, payload)` на реальную запись. `OutboxService.add` пишет строку через переданный `EntityManager` — ту же транзакцию, что и доменная мутация (dual-write исключён структурно, не проверкой). Таблица `outbox` (миграция `1784300000000-CreateOutbox`): `id` uuid (= будущий `messageId`), `routing_key`, `payload` jsonb, `created_at`, `published_at` nullable, `attempts`; частичный индекс `WHERE published_at IS NULL` держит выборку дешёвой независимо от объёма уже опубликованных строк. `OutboxRelay.tick()` — единая Postgres-транзакция на батч: `SELECT ... FOR UPDATE SKIP LOCKED` → publish в `auction.events` (`messageId = row.id`) → `markPublished`, лок держится до конца publish внутри batch-транзакции (осознанно, не publish вне транзакции) — иначе параллельный `tick()` гарантированно перевыбрал бы те же непубликованные строки. Ошибка публикации одной строки ловится построчным try/catch (`recordFailure`, `attempts++`), не откатывая весь батч; падение до коммита batch-транзакции откатывает всё — строки остаются `published_at IS NULL` и переопубликуются на следующем тике (at-least-once, дедуп на консьюмере — M3-01). Тикер — `setInterval` в `onModuleInit`/`onModuleDestroy` с `ticking`-флагом реентерабельности (не `@nestjs/schedule` — не было смысла тянуть зависимость ради одного цикла). Добавлена секция `AppConfigService.outbox` (`pollIntervalMs`, `batchSize`) и env `OUTBOX_POLL_INTERVAL_MS`/`OUTBOX_BATCH_SIZE`.
**Файлы:** `apps/api/src/platform/outbox/{outbox.entity,outbox.repository,outbox.service,outbox.relay,outbox.module}.ts` + `outbox.integration-spec.ts`; `apps/api/src/platform/persistence/migrations/1784300000000-CreateOutbox.ts`; `apps/api/src/config/{env.schema,app-config.service,env.schema.spec}.ts`, `.env.example`.
**Проверено:** `pnpm -C apps/api lint/build/test/test:integration` зелёные. Интеграционные тесты на реальных Postgres+RabbitMQ (testcontainers): (1) атомарность — rollback домена не оставляет строки ни в доменной таблице, ни в outbox; (2) relay публикует ровно по разу помеченные строки, `published_at` проставляется, реальный consumer на `notification.q` получает все сообщения с верным `messageId`; (3) at-least-once — сбой `markPublished` откатывает batch-транзакцию, строка остаётся кандидатом, следующий тик публикует её снова (consumer видит дубль — задокументированное поведение); (4) конкурентность — два независимых инстанса `OutboxRelay` (не два вызова на одном объекте — так тест реально проверяет `FOR UPDATE SKIP LOCKED`, а не флаг реентерабельности) тикают параллельно на общих 5 строках, суммарно ровно 5 доставок без дублей. Ревью `load-auction-reviewer` — **PASS** (0 critical, 0 warning). `spec-guardian` — ALIGNED с §5/§9.5 и контрактом `OutboxPort`. `pattern-verifier` — все DoD-пункты подтверждены прогоном интеграционных тестов.
**Заметки:** security-review отметил один WARNING — `OUTBOX_BATCH_SIZE`/`OUTBOX_POLL_INTERVAL_MS` не имели верхней границы (риск случайно огромного батча/busy-loop от опечатки в конфиге). Исправлено: `.max(1000)`/`.max(60_000)` соответственно, `.min(50)` для интервала. Остальные находки признаны отложенными, не блокирующими: нет верхнего предела на `attempts` (poison-строка ретраится вечно — не усиливает нагрузку на брокер, т.к. `Publisher.publish` не делает внутренних ретраев, но кандидат на dead-letter-парковку в будущем), `payload: jsonb` без ограничения размера (ответственность вызывающего домена, не этого слоя).

## 2026-07-17 — M2-02 platform/messaging: топология RabbitMQ, publisher, base consumer, QoS, retry/DLX

**Что:** Ручная обёртка над `amqplib` + `amqp-connection-manager` (без `@nestjs/microservices`). `TopologyService` декларирует всю топологию §8.1 идемпотентно на каждом (пере)подключении: `auction.events` (topic), `settlement.commands` (direct), `auction.retry`/`auction.dlx` (topic) + очереди `notification.q`/`settlement.q`/`listing.q`/`settlement.steps.q` с биндингами. `Publisher` публикует на confirm-channel с `messageId`, `persistent: true` по умолчанию. `BaseConsumer` — абстрактный класс по шаблону §9.6 (dedup-хук → process → ack), prefetch/QoS на подписке, `retryOrDlq` с экспоненциальным backoff. Retry-механика: при ошибке сообщение публикуется в `auction.retry` с routing key = имя исходной очереди и per-message TTL через `expiration` (не через `x-message-ttl` очереди — так получаем экспоненту без множества уровней очередей); retry-очередь `<name>.retry.q` дедлеттерит через **default exchange** обратно в исходную очередь по имени — не через `auction.events`, иначе ретрай `lot.closed` повторно зафанаутился бы в `notification.q`/`listing.q`. Номер попытки — явный header `x-attempt`, инкрементируемый самим consumer'ом (не `x-death`, которая накапливает неоднозначные записи при многошаговом дедлеттеринге). После `retryLimit` попыток — публикация в `auction.dlx` → `<name>.dlq`. Дедуп-хук — опциональный порт `DedupPort`/`NullDedupPort` (по образцу `OUTBOX_PORT` из M1-03), реальная Redis-реализация — M3-01. Добавлена секция `AppConfigService.messaging` (prefetch, retryLimit, retryBaseTtlMs, retryMultiplier, retryMaxTtlMs) и 5 новых `RABBITMQ_*` env-переменных.
**Файлы:** `apps/api/src/platform/messaging/{amqp-connection.token,messaging.config.token,messaging.constants,topology,publisher,base.consumer,dedup.port,amqp-lifecycle}.ts` + `*.integration-spec.ts`, изменён `messaging.module.ts`; `apps/api/src/config/{env.schema,app-config.service,env.schema.spec}.ts`, `.env.example`; новые зависимости `amqplib`, `amqp-connection-manager`, `@types/amqplib`, `@testcontainers/rabbitmq`.

**Заметки:** security-review нашёл **CRITICAL** уже после первого PASS ревьюера: `JSON.parse` тела сообщения в `base.consumer.ts` вызывался вне try/catch — невалидный JSON приводил к unhandled rejection (т.к. `handle()` вызывается fire-and-forget), что могло уронить процесс и зациклить redelivery, минуя retry/DLQ. Исправлено: `parse()` обёрнут в свой try/catch, ошибка парсинга уходит напрямую в `deadLetterUnparsable` → DLQ с `x-attempt: 1` (минуя retry-цикл — невалидный JSON не станет валидным при повторной попытке), сырое тело сохраняется в DLQ-сообщении для отладки. Заодно закрыт WARNING того же прохода: `RABBITMQ_PREFETCH` получил `.int().min(1)` (0 по семантике AMQP означает "unlimited" и тихо отключил бы backpressure). Повторное ревью после фикса — **PASS**. `spec-guardian` — ALIGNED с §8.1/§9.6/§5/§13. `pattern-verifier` — все DoD-пункты подтверждены прогоном интеграционных тестов, включая новый regression-тест. ## 2026-07-17 — M2-01 platform/redis: Lock, RateLimiter, PubSub, CAS-каркас

**Что:** Расширен `RedisModule` четырьмя примитивами поверх общего `REDIS_CLIENT`: `LockService` (Redlock-lite — `SET NX` + token + Lua compare-token-and-del, плюс `withLock` с авто-release), `RateLimiter` (sliding-window на ZSET через один атомарный Lua-скрипт — `ZREMRANGEBYSCORE`+`ZCARD`+условный `ZADD`/`PEXPIRE`, выбран вместо fixed-counter из-за burst-дыры на границе окна), `PubSub` (мультиплексированный publish/subscribe: один `duplicate()`-коннект на подписку, `Map<channel, Set<handler>>`, ошибка одного handler'а не роняет остальные), `CasService` (каркас `tryBeatHighBid` через `defineCommand`/`evalsha` с формой KEYS/ARGV из §8.3 — заглушка always-accept, сравнение сумм и статуса лота — M3-02). Добавлен `RedisKeys` — единый билдер всех ключей §8.2, чтобы `bidding`/`auction`/будущие пакеты не рассинхронились в написании `lot:{id}:high` (критично для реконсиляции §6). Токен `REDIS_CLIENT` вынесен в отдельный `redis-client.token.ts` без зависимостей — иначе новые сервисы (провайдеры модуля) и сам модуль (импортирующий эти сервисы как провайдеры) образовывали бы циклический импорт, ломающий DI-резолюцию декоратора `@Inject` при подъёме приложения через Nest-контейнер.
**Файлы:** `apps/api/src/platform/redis/{redis-keys,lua-scripts,lock.service,rate-limiter,pub-sub,cas.service,redis-client.token}.ts` + соответствующие `*.spec.ts`/`*.integration-spec.ts`, изменён `redis.module.ts`
**Проверено:** `pnpm -C apps/api lint/build/test/test:integration` зелёные. Интеграционные тесты на реальном Redis (testcontainers): `LockService` — два конкурентных `acquire` одного ключа дают ровно один успешный захват, `release` по верному токену отпускает и разрешает повторный захват, по неверному — отказывает и ключ остаётся занят, TTL истекает и ключ переза­хватывается; `PubSub` — publish доставляет payload подписанному handler'у, после `Unsubscribe()` доставка прекращается; `RateLimiter` — `allowed`/`remaining` корректны внутри окна, блокирует за лимитом, сбрасывается после истечения окна; `CasService` — заглушка пишет `lot:{id}:high` через `evalsha`, подтверждено прямым `HGETALL`. Ревью `load-auction-reviewer` — **PASS** (0 critical, 0 warning). `spec-guardian` — ALIGNED с §8.2/§8.3/§9.4/§5. `pattern-verifier` — все DoD-пункты подтверждены прогоном интеграционных тестов.
**Заметки:** security-review отметил два предмета для будущих задач, не блокирующих M2-01: (1) составной ключ `ratelimit:{carrier}:{lot}` не валидирует входные `carrierId`/`lotId` на спецсимволы — валидация должна лечь на вызывающий код в M3 (DTO ставки, ожидаемо UUID); (2) `lot:{id}:high` пишется без TTL/expire — жизненный цикл этого ключа (истечение вместе с закрытием лота) определяется полной CAS-логикой в M3-02, а не этим каркасом.
