# Worklog

Журнал выполненных задач: что реально сделано, где это лежит и чем доказано. Запись появляется **сразу после того, как ревьюер вернул PASS, и до отчёта о готовности в чате** — это последнее действие задачи, а не первое действие следующей. Команда `/task-done` — лишь один из способов прогнать закрывающий конвейер, но не триггер записи: без неё запись всё равно обязательна. Записи не редактируются и не удаляются: только append сверху.

Задним числом не дописываем. Если обнаружилась выполненная, но не залогированная задача — не латай журнал молча: скажи об этом и прогони закрывающий конвейер (ревью + зелёные lint/build/test), иначе запись будет декларацией, а не фактом.

Назначение — быстрый ответ на «что уже готово и как это проверялось» в начале новой сессии, без раскопок в git log. Статус задач при этом отмечается галочкой в [tasks/INDEX.md](tasks/INDEX.md); детали — здесь.

Формат записи:

```
## YYYY-MM-DD — <ID задачи> <короткое название>
**Что:** 1–3 строки по существу — что появилось, какое решение принято
**Файлы:** ключевые пути (не полный diff)
**Проверено:** тесты/команды, которые это доказывают
**Заметки:** отклонения от задачи, отложенное, замеченные грабли — опционально
```

Правила:

- Одна запись — одна задача из `docs/tasks/`. Задачи вне INDEX.md логируются как `ad-hoc — <название>`.
- Не логируй непроверенное: нет PASS ревьюера или зелёных lint/build/test — нет записи.
- Никакой статистики изменений, никаких чек-листов тест-плана, никаких упоминаний ассистента (см. [code-style.md](../.claude/rules/code-style.md)).
- Уроки и корректировки — не сюда, а в [lessons.md](lessons.md). Здесь — только факт выполнения.

## 2026-07-21 — M4-02 settlement: шаги саги + компенсации (ключевая демонстрация)

**Что:** Оркестрация закрытия лота (§7) поверх каркаса M4-01 — цепочка шагов `lock→winner→reserve→invoice→notify→settle` через RabbitMQ-команды на `settlement.steps.q` (один routing key `settlement.step` на все шаги, различаются телом `{sagaId, lotId, step, direction}`), с компенсациями в обратном порядке при падении. `saga_instances` — источник истины, входящая RMQ-команда — только «пинок»: `SettlementStepConsumer.process()` на каждой доставке сверяет `(step, direction)` из команды с фактическим состоянием саги и, если не совпадает (редоставка, дубль, краш между коммитом и publish), переопубликовывает актуальную команду вместо повторного действия — самоисцеление без outbox для промежуточных шагов (`OutboxRelay` публикует только в `Exchanges.events`, поэтому через outbox идут только терминальные `settlement.completed`/`settlement.failed`, шаги — напрямую через `Publisher`).

Каждый forward- и compensate-шаг — отдельная транзакция: `SagaRepository.lockForUpdate` → повторная проверка `step`/`status` внутри транзакции (защита от гонки двух конкурентных доставок одного шага) → действие → продвижение → commit → публикация следующего кика. Компенсация запускается через `msg.attempt >= retryLimit` (переиспользует штатный backoff `BaseConsumer`, не отдельный счётчик) и идёт в обратном порядке через `previousStep()` (обратная к `nextStep` из M4-01); шаг `winner` не имеет собственной компенсации (нет в таблице §7) — реверс на нём no-op. Redis-лок (шаг 1) держится токеном, персистентным в `saga.payload.lockToken` (генерируется в `SettlementTriggerConsumer`), новый `LockService.acquireOwned` реентерабелен по токену после рестарта процесса; освобождается на успешном `settle` и как часть компенсации шага `lock`. «Нет валидных ставок» переиспользует ту же цепочку компенсаций, стартуя сразу с шага `winner`.

Резерв средств и инвойс — эмуляция как новые таблицы `fund_reservations`/`invoices` (`UNIQUE(lot_id)`, идемпотентные `ON CONFLICT DO NOTHING` вставки, статусы reserved/released и issued/void) — не payload-флаги, а инспектируемые доказательства того, что компенсация реально откатила побочный эффект. Нотификации победителю/шипперу — новые типы `lot_won`/`lot_settled`, `SettlementNotifier` переиспользует `NotificationLogRepository` (M3-06, теперь экспортирован из `NotificationModule`) с детерминированным `messageId` (`settlement:{sagaId}:{type}`, не `randomUUID`), чтобы редоставка шага `notify` не задваивала нотификацию. Ровно один сеттлмент: шаг `settle` явно распознаёт redelivery-после-коммита (`lot.status==='settled' && lot.winningBidId` совпадает) и no-op'ает вместо повторного `transitionLot`, который иначе бросил бы `LotTransitionError` на переходе `settled→settled`.

## 2026-07-21 — M4-01 settlement: модель состояния саги + saga_instances (открывает M4)

**Что:** Новый модуль `modules/settlement` — персистентный каркас оркестрированной саги закрытия лота (§7): таблица `saga_instances` (id, lotId, step, status, payload jsonb, attempts, version, timestamps) + миграция, domain-типы (`SagaStep` lock→winner→reserve→invoice→notify→settle, `SagaStatus` running/compensating/completed/failed, `STEP_ORDER`/`nextStep`), `SagaRepository` (UoW-репозиторий) и `SettlementTriggerConsumer` — идемпотентный consumer на уже существующей с M2-02 `settlement.q` (`lot.closed`), создающий сагу в статусе `running` на шаге `lock`. Шаги саги и компенсации — не в объёме, это M4-02; `settlement.commands`/`settlement.steps.q` объявлены с M2-02, но потребителя пока нет.

Идемпотентность создания — двойная защита: `BaseConsumer`-дедуп по `(queue, messageId)` плюс `UNIQUE(lot_id)` на таблице + `ON CONFLICT DO NOTHING` (одна сага на лот в принципе, независимо от messageId редоставленного или republish-нутого события) — тот же паттерн, что `NotificationLogRepository.record`.

**Найденный по ходу нюанс TypeORM (задокументирован, не блокирует):** `SagaRepository.update()` зеркалит `LotRepository.update()` — `repo(tx).save(...)`. TypeORM `Repository#save()` не бросает ошибку optimistic-lock из устаревшего in-memory объекта (только `QueryBuilder#setLock('optimistic', v)` на чтении это делает) — `@VersionColumn` инкрементится только на свежепрочитанном объекте, стейл-версия записывается как есть без CAS-проверки. `LotRepository` не задевает эту яму только потому, что его вызывающие всегда делают `lockForUpdate()`+`save()` в одной транзакции. Пока `update()` нигде не вызывается (единственный писатель — `create()` в триггере), риска нет; для M4-02 (несколько step-консьюмеров будут двигать одну сагу) это открытый вопрос — оставлен явный комментарий на `update()` и интеграционный тест, доказывающий фактическое поведение.

## 2026-07-21 — M3-06 notification: идемпотентный мультиканальный консьюмер (закрывает M3)

**Что:** Consumer на `notification.q` (забинджена ещё с M2-02 на `bid.placed`/`lot.opened`/`lot.closed`, но потребителя не было). Два канала: realtime-push и email-stub. Последняя задача milestone M3 — весь горячий путь (idempotency→CAS→bidding→realtime→notification) закрыт.

Ключевое архитектурное разграничение: M3-05 уже транслирует эти три события всем WS-клиентам комнаты лота (generic broadcast) — notification-модуль не дублирует это, а добавляет то, чего M3-05 в принципе не может: персональные уведомления конкретным людям (кого перебили — предыдущему лидеру ставки; шипперу — о новой активности/закрытии). "Кого перебили" — новый `BidRepository.findPreviousBest(lotId, excludeBidId)`: лучшая ставка на лот, исключая только что принятую — поскольку `bid.placed` публикуется только для реально принятой (CAS уже отсеял всё остальное), исключённая ставка гарантированно была текущим минимумом, а лучшая среди оставшихся — это и есть "лидировал до этой ставки". Самоперебитие (carrier улучшает свою же лидирующую заявку) не порождает outbid-уведомление о самом себе. Победитель/шиппер на `lot.closed` — не в объёме (придёт из saga в M4, как явно оговорено в тексте задачи и §7 ТЗ) — здесь только generic "лот закрылся" шипперу.

Email-stub — таблица `notifications` с составным unique-ключом `(message_id, recipient_id, type, channel)` + `INSERT ... ON CONFLICT DO NOTHING`. Понадобилось отдельно от дедупа на уровне очереди: `BaseConsumer.dedup.mark()` срабатывает только после успеха ВСЕГО `process()` — частичный сбой (например realtime прошёл, email упал) откатывает retry всего `process()` целиком, и без собственной идемпотентности на уровне БД запись задублировалась бы (или упала на unique-violation без constraint).

**Найден и исправлен реальный баг** (сам implementing-агент, до ревью): `NotificationEntity` изначально не имела `@Unique(...)`-декоратора — миграция создаёт constraint через raw SQL, но интеграционные тесты используют `synchronize:true` (общий паттерн всего проекта), которое НЕ применяет миграции — значит `.orIgnore()` в тестах был no-op (не на что ссылаться), и повторная доставка реально дублировала строки. Добавлен `@Unique(['messageId','recipientId','type','channel'])` на entity — совпадает по колонкам с миграцией (разные только автосгенерированные/явные имена constraint, что не важно для `ON CONFLICT DO NOTHING` без имени target). Подтверждено ревью как корректный и load-bearing фикс.

## 2026-07-21 — M3-05 realtime: WS-gateway + Redis Pub/Sub fan-out

**Что:** Первая WS-фича проекта — `WS /realtime` на `@nestjs/websockets`+`socket.io` (новые зависимости), фанаут событий лота (`bid.placed`, `lot.opened/closing/closed/cancelled`, `settlement.completed/failed`) клиентам, подписанным на канал лота, устойчивый к нескольким инстансам API через Redis Pub/Sub (`platform/redis/pub-sub.ts`, M2-01, использован как есть). Доставка — через мост `RealtimeBridgeConsumer` (новая очередь `realtime.q`), не прямой publish из `PlaceBidHandler`: канонический путь §6 (`outbox relay → RabbitMQ → gateway → WS`) гарантирует фанаут только реально закоммиченного, а не состояния, которое Postgres TX может ещё откатить после CAS-успеха. Фанаут между инстансами — только через ручной Redis Pub/Sub (уже готовый примитив M2-01), без `@socket.io/redis-adapter` — готовые обёртки такого рода проект намеренно не берёт.

По ходу закрыта дыра в топологии: `lot.closing` публиковался `CloseLotHandler` с M2-06, но не был забинджен ни к одной очереди — событие проваливалось в никуда. **Найдены и исправлены два реальных concurrency-бага (reviewer):** (1) race в ref-counting Pub/Sub-подписок gateway — `acquire()` проверял-и-действовал через `await`, из-за чего два клиента, одновременно подписывающиеся на ещё не отслеживаемый лот, могли оба запустить свою Redis-подписку: первая утекала (никогда не отписывалась), а событие доставлялось клиентам дважды. Исправлено резервированием слота в `Map` синхронно, до `await` на сам `subscribe()`. (2) TOCTOU в аутентификации — `onSubscribe`/`onUnsubscribe` не проверяли `client.data.user`, полагаясь только на то, что `handleConnection` успеет разорвать неаутентифицированное соединение раньше; добавлена явная проверка в обоих хендлерах.

**Security-review** также нашёл: throttling (`ThrottlerGuard`, глобальный `APP_GUARD`) физически не доходит до WS-хендлеров (эмпирически подтверждено — 0 вызовов на реальном соединении) — в сочетании с неограниченным числом `subscribe` на клиента это грозило неограниченным ростом Redis-подписок на одном общем subscriber-соединении (shared-fate ресурс для всех клиентов процесса). Добавлен `MAX_LOTS_PER_SOCKET=50`. Также добавлена валидация `lotId` в `RealtimeBridgeConsumer.process` — payload без `lotId` (например будущий settlement-евент без этого поля) раньше тихо публиковался бы в канал `lot:undefined:channel`; теперь логируется и пропускается.

## 2026-07-21 — M3-03 bidding: горячий путь PlaceBid (idem → CAS → TX+outbox)

**Что:** Новый модуль `modules/bidding` (domain/infrastructure/application/api) — центральный демонстрационный путь: `POST /lots/:lotId/bids` → rate-limit → идемпотентность → Lua CAS → Postgres TX+outbox → `bid.placed`. Собирает воедино примитивы M3-01/M3-02/M2-03, ничего в них не меняя (кроме одного нового read-метода в `LotRepository`).

Архитектурные решения: (1) `lot.version` НЕ бампается на каждую ставку — порядок ставок целиком решает атомарный Lua CAS на `lot:{id}:high`, версия лота — только для state-machine жизненного цикла; бамп на каждый bid создал бы write-contention на единственную строку `lots` и коллизии с `CloseLotHandler`'s `FOR UPDATE`. (2) `LotRepository.readStatus` — non-locking MVCC-чтение статуса лота внутри TX ставки (не `lockForUpdate`) — defense-in-depth против Redis/Postgres дрейфа (§6 "тонкое место") без блокировок. (3) Reconciliation на падении TX после успешного CAS — `BidRepository.findCurrentBest` + `cas.reconcileIfCurrent(lotId, bidId, best)` (race-safe примитив из M3-02, fenced по bidId); `LotNotOpenError` → закэшированный `closed`-reject, прочие ошибки → rethrow без `idem.complete` (маркер `in_progress` истечёт по TTL). (4) Порядок шагов: rate-limit → idempotency → CAS → TX → complete — rate-limit первым, иначе 429 либо "утекает" idem-маркер, либо кэшируется на `doneTtlMs` без способа снять (`idem.abort` не существует). (5) `Money` продублирован в `bidding/domain` — нет прецедента межмодульного domain-импорта. (6) Миграция `bids`: без FK на `lot_id` (INSERT в дочернюю таблицу берёт `FOR KEY SHARE` на родительскую строку — конфликтовало бы с `FOR UPDATE` в `CloseLotHandler`), без unique на `idempotency_key` (ключ в Redis скоуплен `${carrierId}:${idempotencyKey}`, два carrier легитимно переиспользуют одну строку заголовка; голая БД-уникальность превращала бы post-TTL replay в constraint-violation 500).

**Найдено ревью и исправлено:** контроллер изначально навешивал `@UsePipes(new ValidationPipe({errorHttpStatusCode:422}))` в расчёте на коды `201/409/422` из текста задачи — но это мёртвый код: Nest цепляет пайпы в порядке global→class→method для одного и того же параметра, и глобальный `ValidationPipe` из `main.ts` (400 по умолчанию) выполняется первым и рвёт цепочку своим исключением раньше, чем управление доходит до controller-scoped пайпа (подтверждено чтением исходников `@nestjs/core/pipes/pipes-consumer.js`/`context-creator.js`). 422 недостижим в принципе при уже существующем глобальном пайпе — не баг реализации, а архитектурное ограничение Nest. Решение: убрать мёртвый controller-scoped пайп, стандартизировать на 400 (совпадает с конвенцией остального приложения, например `CreateLotDto`) — сознательное отклонение от буквального текста задачи (`201/409/422` → `201/409/429/400`), а не недосмотр.

**Проверено и отклонено:** ревью (reviewer) также заявил, что `@Post()` без `@HttpCode` по умолчанию отдаёт 200, а не 201 — проверено по факту в исходниках `@nestjs/core/router/router-response-controller.js` (`getStatusByMethod` возвращает `HttpStatus.CREATED` именно для `POST`) — заявление ложное, правка не потребовалась.

**Файлы:** `apps/api/src/modules/bidding/**` (новый модуль целиком: `domain/{bid,errors}.ts`, `infrastructure/{bid.entity,bid.mapper,bid.repository}.ts`, `application/place-bid.handler.ts` (+ `.spec.ts`, `bidding-hot-path.integration-spec.ts`), `api/{bids.controller,dto/place-bid.dto}.ts`, `bidding.module.ts`); `apps/api/src/platform/persistence/migrations/1784600000000-CreateBids.ts`; `apps/api/src/modules/auction/infrastructure/lot.repository.ts` (+ `readStatus`); `apps/api/src/config/{env.schema,app-config.service}.ts` (секция `bidding`: `BID_RATE_LIMIT`/`BID_RATE_WINDOW_MS`); `apps/api/src/app.module.ts` (регистрация `BiddingModule`).

## 2026-07-20 — M3-02 Lua CAS: атомарный compare-and-set лучшей ставки + reconciliation

**Что:** Заменена заглушка `CAS_BEAT_HIGH_BID` (безусловный `HSET`, существовала с M2-01) на настоящую §8.3-логику: status-gate (`lot:{id}:status`) + reverse-auction сравнение (`>=` отклоняется, меньше — принимается) в одном атомарном Lua-скрипте. Плюс защита от `NaN`: `String(amount)` для `NaN`/`Infinity` не падает в `tonumber()` — конвертируется в реальный Lua-float, а `NaN` в частности ломает сравнение навсегда (`cur >= NaN` всегда `false`, значит любая следующая ставка проходила бы CAS без проверки) — добавлен explicit `amount ~= amount`-guard (стандартный Lua-идиом на NaN) перед сравнением, дешёвая защита на platform-уровне вместо того, чтобы полагаться только на будущую DTO-валидацию в M3-03 (найдено security-review).

Reconciliation (§6, "тонкое место": Redis high-bid — кандидат, Postgres — источник истины) — два разных Redis-примитива на `CasService`, а не один, как буквально предполагал текст задачи: `reconcileIfCurrent(lotId, expectedBidId, candidate|null)` — race-safe компенсация для rollback-ветки (атомарный Lua compare-token-then-act по `bidId`, паттерн как у существующего `LOCK_RELEASE`) — если между чтением БД и вызовом reconcile пришла реально более новая ставка, `bidId` в Redis уже другой, fence не пройдёт, новый кандидат не будет затёрт; и `reconcile(lotId, candidate|null)` — безусловная перезапись/очистка для cold-start/open-init, где конкурентного писателя нет по построению. Причина разделения: единая безусловная функция могла бы затереть легитимно более новую ставку, если БД-чтение обгоняет следующий CAS — сравнение по amount не отличает "призрака" от реально более новой ставки (обе "ниже"), только fence по `bidId` race-safe на уровне Redis без токена версии.

Таблицы `bids`/модуля `bidding` ещё нет (M3-03 зависит от M3-02) — оркестрация "прочитать факт из БД → вызвать reconcile" в M3-02 не реализуется (только сам примитив, `platform/*` и так не может импортировать `modules/*`), а DoD-тест на reconciliation после потерянного коммита подставляет факт напрямую, доказывая механизм на уровне `CasService`, без ожидания реального вызывающего (тот же паттерн, что и `IdempotencyService` в M3-01 — сначала примитив, реальный вызывающий — M3-03).

`OpenLotHandler`: добавлен `cas.reconcile(lotId, null)` перед `cas.setStatus(lotId, 'open')` — защита от протухшего `lot:{id}:high` (например, от предыдущего незавершённого прогона); порядок важен — лот гарантированно ещё не `open` в момент очистки, поэтому окна для CAS против недочищенного хэша нет. `CloseLotHandler`/`CancelLotHandler` не менялись (high на close не чистится намеренно — статус `closing` уже блокирует новые ставки через CAS-gate; cancel уже чистит оба ключа через `cas.clear`).

TTL на `lot:{id}:high`/`lot:{id}:status` решено не добавлять (вопрос был открыт с M2-01): ключи управляются жизненным циклом лота, а не временем — TTL на status мог бы ложно перевести долгоживущий (anti-snipe-продлённый) открытый лот в `closed`-ветку, TTL на high мог бы уронить легитимного лидера посреди аукциона. Очистка после `settled` — забота саги сеттлмента (M4), вне объёма этой задачи.

**Что:** Идемпотентность на двух уровнях (§5 ТЗ), без сторонних библиотек — только `ioredis`. `IdempotencyService.begin/complete` — единственный механизм кэширования; вызывается explicit из хендлеров (первый реальный вызывающий — `PlaceBidHandler` в M3-03). Ключевое архитектурное решение: текст задачи описывал "API-интерсептор, оборачивающий горячие POST" (прозрачный HTTP-кэш), но §9.3 ТЗ (skeleton `PlaceBidHandler`) показывает другой паттерн — `begin()`/`complete()` explicit внутри хендлера, между которыми идут Lua CAS и Postgres TX; интерсептор физически не может выразить этот порядок без знания домена. Следуя ТЗ как источнику истины, "интерсептор" схлопнут в тонкий `RequireIdempotencyKeyGuard` + `@IdempotencyKey()`-декоратор — только валидируют заголовок, кэширования не делают. Модуля `bidding`/`POST /lots/:id/bids` ещё нет (M3-03), поэтому подключить guard к реальному контроллеру сейчас не на чем — M3-01 поставляет готовый примитив.

`begin(key)` — один атомарный Lua-скрипт `IDEM_BEGIN` (GET-или-SET-с-TTL за один round-trip, без окна гонки между `SET NX` и `GET`), возвращает `new`/`replay`/`in_progress` (три состояния, а не два из буквального skeleton'а §9.3 — `in_progress` нужен для DoD "параллельные дубли не выполняют операцию дважды"). `complete(key, result)` перезаписывает `idem:{key}` со свежим (более долгим) TTL — короткая in-progress-аренда сменяется окном повтора.

Консьюмерский дедуп (`RedisDedupPort`) — `seen()` read-only (`EXISTS`), `mark()` только после успешного `process()` в `BaseConsumer`: резервирование в `seen()` сломало бы retry (`retryOrDlq` переопубликует с тем же `messageId`, и он был бы ложно задедуплен). Раньше это маскировалось `NullDedupPort`. Исправлено: ключ и весь контракт (`RedisKeys.msgDedup`, `DedupPort.seen/mark`, `RedisDedupPort`, `BaseConsumer`) теперь скоупятся по паре `(queue, messageId)` → `msg:dedup:{queue}:{messageId}`. §8.2 ТЗ обновлён под фактическое поведение с пояснением. Проверено интеграционным тестом с тремя живыми консьюмерами на трёх реальных очередях топологии.

## 2026-07-17 — M2-06 auction: команды CreateLot/OpenLot/CloseLot/CancelLot + scheduler + outbox

## 2026-07-17 — M2-04 platform/scheduler: ZSET-планировщик отложенных команд

**Что:** Планировщик отложенного open/close лота поверх Redis ZSET (`auction:schedule:open`/`auction:schedule:close`). Отклонение от буквального скелета §9.7: вместо раздельных `zrangebyscore`→dispatch→`zrem` (три шага с race-окном — два параллельных тика увидели бы один и тот же элемент до его удаления) — `ZSetScheduler.tick()` вызывает Lua-скрипт `SCHEDULER_CLAIM_DUE`, атомарно совмещающий `ZRANGEBYSCORE`+`ZREM` в одной операции; `dispatch()` вызывается уже после атомарного claim, вне Lua. `schedule(setKey, dueAtMs, payload)` — обычный `ZADD`; `payload` обязан быть стабильным идентификатором (лотId) — повторный `schedule` того же payload обновляет score существующего члена (продление/анти-снайп), а не создаёт дубликат — это задокументировано как инвариант в JSDoc. При ошибке `dispatch()` после claim элемент не теряется — `ZADD GT` возвращает его в ZSET с задержкой (`retryDelayMs`); флаг `GT` не откатывает более позднюю дату, если конкурентный `schedule()` успел продлить срок в промежутке. `SchedulerTicker` — `setInterval` в `onModuleInit`/`onModuleDestroy` с `ticking`-флагом реентерабельности (по образцу `OutboxRelay`), на каждом цикле тикает оба ключа (open/close) через опциональный порт `SchedulerDispatchPort` (throwing `NullSchedulerDispatchPort` до будущей проводки auction-модуля — фолбэк создаётся внутри конструктора `SchedulerTicker` через `@Optional()`, не как провайдер модуля, чтобы будущий модуль мог переопределить токен без конфликта).
