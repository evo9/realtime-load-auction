# Worklog

Журнал выполненных задач: что реально сделано, где это лежит и чем доказано. Запись появляется **сразу после того, как ревьюер вернул PASS, и до отчёта о готовности в чате** — это последнее действие задачи, а не первое действие следующей. Команда `/task-done` — лишь один из способов прогнать закрывающий конвейер, но не триггер записи: без неё запись всё равно обязательна. Записи не редактируются и не удаляются: только append сверху.

Задним числом не дописываем. Если обнаружилась выполненная, но не залогированная задача — не латай журнал молча: скажи об этом и прогони закрывающий конвейер (ревью + зелёные lint/build/test), иначе запись будет декларацией, а не фактом.

Назначение — быстрый ответ на «что уже готово и как это проверялось» в начале новой сессии, без раскопок в git log. Статус задач при этом отмечается галочкой в [tasks/INDEX.md](tasks/INDEX.md); детали — здесь.

Формат записи:

```
## YYYY-MM-DD — <ID задачи> <короткое название>
**Что:** 1–3 строки по существу — что появилось, какое решение принято
**Файлы:** ключевые пути (не полный diff)
**Проверено:** тесты/команды, которые это доказывают
**Заметки:** отклонения от задачи, отложенное, замеченные грабли — опционально
```

Правила:

- Одна запись — одна задача из `docs/tasks/`. Задачи вне INDEX.md логируются как `ad-hoc — <название>`.
- Не логируй непроверенное: нет PASS ревьюера или зелёных lint/build/test — нет записи.
- Никакой статистики изменений, никаких чек-листов тест-плана, никаких упоминаний ассистента (см. [code-style.md](../.claude/rules/code-style.md)).
- Уроки и корректировки — не сюда, а в [lessons.md](lessons.md). Здесь — только факт выполнения.

## 2026-07-21 — M3-03 bidding: горячий путь PlaceBid (idem → CAS → TX+outbox)

**Что:** Новый модуль `modules/bidding` (domain/infrastructure/application/api) — центральный демонстрационный путь: `POST /lots/:lotId/bids` → rate-limit → идемпотентность → Lua CAS → Postgres TX+outbox → `bid.placed`. Собирает воедино примитивы M3-01/M3-02/M2-03, ничего в них не меняя (кроме одного нового read-метода в `LotRepository`).

Архитектурные решения: (1) `lot.version` НЕ бампается на каждую ставку — порядок ставок целиком решает атомарный Lua CAS на `lot:{id}:high`, версия лота — только для state-machine жизненного цикла; бамп на каждый bid создал бы write-contention на единственную строку `lots` и коллизии с `CloseLotHandler`'s `FOR UPDATE`. (2) `LotRepository.readStatus` — non-locking MVCC-чтение статуса лота внутри TX ставки (не `lockForUpdate`) — defense-in-depth против Redis/Postgres дрейфа (§6 "тонкое место") без блокировок. (3) Reconciliation на падении TX после успешного CAS — `BidRepository.findCurrentBest` + `cas.reconcileIfCurrent(lotId, bidId, best)` (race-safe примитив из M3-02, fenced по bidId); `LotNotOpenError` → закэшированный `closed`-reject, прочие ошибки → rethrow без `idem.complete` (маркер `in_progress` истечёт по TTL). (4) Порядок шагов: rate-limit → idempotency → CAS → TX → complete — rate-limit первым, иначе 429 либо "утекает" idem-маркер, либо кэшируется на `doneTtlMs` без способа снять (`idem.abort` не существует). (5) `Money` продублирован в `bidding/domain` — нет прецедента межмодульного domain-импорта. (6) Миграция `bids`: без FK на `lot_id` (INSERT в дочернюю таблицу берёт `FOR KEY SHARE` на родительскую строку — конфликтовало бы с `FOR UPDATE` в `CloseLotHandler`), без unique на `idempotency_key` (ключ в Redis скоуплен `${carrierId}:${idempotencyKey}`, два carrier легитимно переиспользуют одну строку заголовка; голая БД-уникальность превращала бы post-TTL replay в constraint-violation 500).

**Найдено ревью и исправлено:** контроллер изначально навешивал `@UsePipes(new ValidationPipe({errorHttpStatusCode:422}))` в расчёте на коды `201/409/422` из текста задачи — но это мёртвый код: Nest цепляет пайпы в порядке global→class→method для одного и того же параметра, и глобальный `ValidationPipe` из `main.ts` (400 по умолчанию) выполняется первым и рвёт цепочку своим исключением раньше, чем управление доходит до controller-scoped пайпа (подтверждено чтением исходников `@nestjs/core/pipes/pipes-consumer.js`/`context-creator.js`). 422 недостижим в принципе при уже существующем глобальном пайпе — не баг реализации, а архитектурное ограничение Nest. Решение: убрать мёртвый controller-scoped пайп, стандартизировать на 400 (совпадает с конвенцией остального приложения, например `CreateLotDto`) — сознательное отклонение от буквального текста задачи (`201/409/422` → `201/409/429/400`), а не недосмотр.

**Проверено и отклонено:** ревью (reviewer) также заявил, что `@Post()` без `@HttpCode` по умолчанию отдаёт 200, а не 201 — проверено по факту в исходниках `@nestjs/core/router/router-response-controller.js` (`getStatusByMethod` возвращает `HttpStatus.CREATED` именно для `POST`) — заявление ложное, правка не потребовалась.

**Файлы:** `apps/api/src/modules/bidding/**` (новый модуль целиком: `domain/{bid,errors}.ts`, `infrastructure/{bid.entity,bid.mapper,bid.repository}.ts`, `application/place-bid.handler.ts` (+ `.spec.ts`, `bidding-hot-path.integration-spec.ts`), `api/{bids.controller,dto/place-bid.dto}.ts`, `bidding.module.ts`); `apps/api/src/platform/persistence/migrations/1784600000000-CreateBids.ts`; `apps/api/src/modules/auction/infrastructure/lot.repository.ts` (+ `readStatus`); `apps/api/src/config/{env.schema,app-config.service}.ts` (секция `bidding`: `BID_RATE_LIMIT`/`BID_RATE_WINDOW_MS`); `apps/api/src/app.module.ts` (регистрация `BiddingModule`).

**Проверено:** `pnpm -C apps/api lint/build/test` зелёные (106 unit-тестов). `test:integration` — 21/21 сьютов, 59/59 тестов (реальные Postgres/Redis, testcontainers), существующие auction/CAS/idempotency/outbox integration-специи не задеты. DoD подтверждён интеграционными тестами: (1) валидная лучшая ставка → accepted, строка в `bids`, outbox-строка `bid.placed`, `lot:{id}:high` обновлён; (2) хуже текущей/лот closing → rejected без новой строки; (3) повтор с тем же scoped-ключом → идентичный ответ, одна строка в `bids`; (4) конкурентные ставки — globally-lowest побеждает, БД и Redis согласованы; (5) rate-limit режет всплеск; доп. — reconciliation после искусственного сбоя TX внутри полного хендлера (не только изолированно на `CasService`, как в M3-02), и межперевозчиковая изоляция идемпотентности при переиспользовании одного `Idempotency-Key`. `reviewer` — PASS WITH WARNINGS (422/400-баг найден и исправлен, ложная находка про 201/200 отклонена после проверки исходников Nest). `spec-guardian` — ALIGNED (единственное расхождение — то же ложное 200/201-утверждение, отклонено). `security-review` — существенных находок нет; закрыты обе находки предыдущих задач (idempotency-key scoping по carrierId — подтверждено кодом; NaN-guard в Lua — подтверждено, что DTO теперь закрывает HTTP-путь, Lua-guard остаётся defense-in-depth для будущих non-HTTP вызывающих); один Low-пробел в тестовом покрытии (сборка scoped-ключа, межперевозчиковая изоляция) — закрыт двумя добавленными тестами. `pattern-verifier` — паттерн доказан на реальной инфре, два независимых прогона без флейка.

## 2026-07-20 — M3-02 Lua CAS: атомарный compare-and-set лучшей ставки + reconciliation

**Что:** Заменена заглушка `CAS_BEAT_HIGH_BID` (безусловный `HSET`, существовала с M2-01) на настоящую §8.3-логику: status-gate (`lot:{id}:status`) + reverse-auction сравнение (`>=` отклоняется, меньше — принимается) в одном атомарном Lua-скрипте. Плюс защита от `NaN`: `String(amount)` для `NaN`/`Infinity` не падает в `tonumber()` — конвертируется в реальный Lua-float, а `NaN` в частности ломает сравнение навсегда (`cur >= NaN` всегда `false`, значит любая следующая ставка проходила бы CAS без проверки) — добавлен explicit `amount ~= amount`-guard (стандартный Lua-идиом на NaN) перед сравнением, дешёвая защита на platform-уровне вместо того, чтобы полагаться только на будущую DTO-валидацию в M3-03 (найдено security-review).

Reconciliation (§6, "тонкое место": Redis high-bid — кандидат, Postgres — источник истины) — два разных Redis-примитива на `CasService`, а не один, как буквально предполагал текст задачи: `reconcileIfCurrent(lotId, expectedBidId, candidate|null)` — race-safe компенсация для rollback-ветки (атомарный Lua compare-token-then-act по `bidId`, паттерн как у существующего `LOCK_RELEASE`) — если между чтением БД и вызовом reconcile пришла реально более новая ставка, `bidId` в Redis уже другой, fence не пройдёт, новый кандидат не будет затёрт; и `reconcile(lotId, candidate|null)` — безусловная перезапись/очистка для cold-start/open-init, где конкурентного писателя нет по построению. Причина разделения: единая безусловная функция могла бы затереть легитимно более новую ставку, если БД-чтение обгоняет следующий CAS — сравнение по amount не отличает "призрака" от реально более новой ставки (обе "ниже"), только fence по `bidId` race-safe на уровне Redis без токена версии.

Таблицы `bids`/модуля `bidding` ещё нет (M3-03 зависит от M3-02) — оркестрация "прочитать факт из БД → вызвать reconcile" в M3-02 не реализуется (только сам примитив, `platform/*` и так не может импортировать `modules/*`), а DoD-тест на reconciliation после потерянного коммита подставляет факт напрямую, доказывая механизм на уровне `CasService`, без ожидания реального вызывающего (тот же паттерн, что и `IdempotencyService` в M3-01 — сначала примитив, реальный вызывающий — M3-03).

`OpenLotHandler`: добавлен `cas.reconcile(lotId, null)` перед `cas.setStatus(lotId, 'open')` — защита от протухшего `lot:{id}:high` (например, от предыдущего незавершённого прогона); порядок важен — лот гарантированно ещё не `open` в момент очистки, поэтому окна для CAS против недочищенного хэша нет. `CloseLotHandler`/`CancelLotHandler` не менялись (high на close не чистится намеренно — статус `closing` уже блокирует новые ставки через CAS-gate; cancel уже чистит оба ключа через `cas.clear`).

TTL на `lot:{id}:high`/`lot:{id}:status` решено не добавлять (вопрос был открыт с M2-01): ключи управляются жизненным циклом лота, а не временем — TTL на status мог бы ложно перевести долгоживущий (anti-snipe-продлённый) открытый лот в `closed`-ветку, TTL на high мог бы уронить легитимного лидера посреди аукциона. Очистка после `settled` — забота саги сеттлмента (M4), вне объёма этой задачи.

**Файлы:** `apps/api/src/platform/redis/lua-scripts.ts` (`CAS_BEAT_HIGH_BID` — реальная логика + NaN-guard; новый `RECONCILE_IF_CURRENT`); `apps/api/src/platform/redis/cas.service.ts` (`HighBidCandidate`, `reconcileIfCurrent`, `reconcile`); `apps/api/src/platform/redis/cas.service.integration-spec.ts` (расширен); `apps/api/src/modules/auction/application/open-lot.handler.ts` (+ `.spec.ts`).

**Проверено:** `pnpm -C apps/api lint/build/test` зелёные (97 unit-тестов). `test:integration` — 20/20 сьютов, 52/52 теста (реальный Redis/Postgres/RabbitMQ, testcontainers), прогонялся неоднократно (включая отдельно `cas.service.integration-spec.ts` дважды подряд) — стабилен, без флейка. Существующие `auction-lifecycle`/`auction-close-concurrency`/`close-lot.anti-snipe` integration-специи остались зелёными без изменений — новая CAS-логика их не задела (они трогают только `lot:{id}:status`, не `high`). DoD подтверждён: (i) `Promise.all` из 6 конкурентных ставок с разными суммами на реальном Redis → побеждает только глобально лучшая; (ii) отсутствующий/`closing` статус → `closed`, равная/большая сумма → `too_low` (отдельные тесты); (iii) reconciliation после потерянного коммита восстанавливает факт из БД (`reconcileIfCurrent`, включая fence-safety — не затирает более новую легитимную ставку). `reviewer` (load-auction-reviewer) — **PASS**, 0 critical/warning. `spec-guardian` — **ALIGNED** (Lua-логика и сигнатура `CasService` — точное совпадение с §8.3/§9.4, reconciliation — обоснованное расширение §6-принципа, не отклонение). `security-review` — 1 находка (integrity-риск через `NaN` в `amount`, LOW/недостижимо без реального вызывающего) устранена Lua-guard'ом и покрыта тестом; остальное (Lua-инъекция через ARGV, hash-field injection, `lotId` key-shape) — без находок. `pattern-verifier` — паттерн доказан на реальной инфре, два независимых прогона без флейка.

**Заметки:** Задача в исходном тексте предполагала одну функцию reconciliation — архитектор спроектировал две (`reconcileIfCurrent`/`reconcile`) с обоснованием race-safety; это уточнение §6-принципа, а не отклонение от него (подтверждено spec-guardian). У `reconcileIfCurrent` пока нет реального вызывающего — контракт для `PlaceBidHandler` в M3-03 (rollback-ветка после CAS + Postgres TX), задокументирован в JSDoc на `CasService`. Отложено (security-review, не блокирует): валидация `amount` (`@IsNumber({allowNaN:false, allowInfinity:false}) @IsPositive()`) должна появиться на уровне DTO в M3-03 — Lua-guard на NaN закрывает вопрос на platform-уровне уже сейчас, но не заменяет валидацию входа.

**Что:** Идемпотентность на двух уровнях (§5 ТЗ), без сторонних библиотек — только `ioredis`. `IdempotencyService.begin/complete` — единственный механизм кэширования; вызывается explicit из хендлеров (первый реальный вызывающий — `PlaceBidHandler` в M3-03). Ключевое архитектурное решение: текст задачи описывал "API-интерсептор, оборачивающий горячие POST" (прозрачный HTTP-кэш), но §9.3 ТЗ (skeleton `PlaceBidHandler`) показывает другой паттерн — `begin()`/`complete()` explicit внутри хендлера, между которыми идут Lua CAS и Postgres TX; интерсептор физически не может выразить этот порядок без знания домена. Следуя ТЗ как источнику истины, "интерсептор" схлопнут в тонкий `RequireIdempotencyKeyGuard` + `@IdempotencyKey()`-декоратор — только валидируют заголовок, кэширования не делают. Модуля `bidding`/`POST /lots/:id/bids` ещё нет (M3-03), поэтому подключить guard к реальному контроллеру сейчас не на чем — M3-01 поставляет готовый примитив.

`begin(key)` — один атомарный Lua-скрипт `IDEM_BEGIN` (GET-или-SET-с-TTL за один round-trip, без окна гонки между `SET NX` и `GET`), возвращает `new`/`replay`/`in_progress` (три состояния, а не два из буквального skeleton'а §9.3 — `in_progress` нужен для DoD "параллельные дубли не выполняют операцию дважды"). `complete(key, result)` перезаписывает `idem:{key}` со свежим (более долгим) TTL — короткая in-progress-аренда сменяется окном повтора.

Консьюмерский дедуп (`RedisDedupPort`) — `seen()` read-only (`EXISTS`), `mark()` только после успешного `process()` в `BaseConsumer`: резервирование в `seen()` сломало бы retry (`retryOrDlq` переопубликует с тем же `messageId`, и он был бы ложно задедуплен). Единственная правка `BaseConsumer` — `seen()` перенесён внутрь `try` (был снаружи с M2-02): сбой Redis в проверке теперь идёт в `retryOrDlq`, а не роняет `handle()` необработанным.

**Найденный по ходу баг (не в исходном плане):** `msg:dedup:{messageId}` глобально по `messageId` ломает fan-out — `lot.closed` биндится сразу на `notification.q`/`settlement.q`/`listing.q` с ОДНИМ `messageId` (outbox публикует один раз на строку), и первый консьюмер, отметивший ключ, "съедал" бы событие для двух остальных очередей. Раньше это маскировалось `NullDedupPort`. Исправлено: ключ и весь контракт (`RedisKeys.msgDedup`, `DedupPort.seen/mark`, `RedisDedupPort`, `BaseConsumer`) теперь скоупятся по паре `(queue, messageId)` → `msg:dedup:{queue}:{messageId}`. §8.2 ТЗ обновлён под фактическое поведение с пояснением. Проверено интеграционным тестом с тремя живыми консьюмерами на трёх реальных очередях топологии.

**Файлы:** `apps/api/src/platform/idempotency/{idempotency.service,redis-dedup.port,idempotency-key.decorator,require-idempotency-key.guard,idempotency.module}.ts` (+ `*.spec.ts`/`*.integration-spec.ts`); `apps/api/src/platform/redis/lua-scripts.ts` (`IDEM_BEGIN`), `redis-keys.ts` (`msgDedup(queue, messageId)`); `apps/api/src/platform/messaging/{base.consumer,dedup.port,messaging.module}.ts`, `dedup.integration-spec.ts`; `apps/api/src/config/{env.schema,app-config.service}.ts` (секция `idempotency`: `IDEMPOTENCY_INPROGRESS_TTL_MS`/`IDEMPOTENCY_DONE_TTL_MS`/`MSG_DEDUP_TTL_MS`); `apps/api/src/app.module.ts` (pino `redact` для заголовка `Idempotency-Key`); `docs/specs/load-auction-spec.md` (§8.2 — уточнён формат ключа дедупа).

## 2026-07-17 — M2-06 auction: команды CreateLot/OpenLot/CloseLot/CancelLot + scheduler + outbox

**Что:** Write-путь модуля `auction` (§4). Хендлеры — обычные `@Injectable` (без CommandBus): `CreateLot` создаёт лот сразу в статусе `scheduled` (физический `draft` не наблюдаем — не пишется отдельной строкой, `draft` остаётся в state-machine для возможного будущего "черновика") и планирует ОБА таймера (`open`+`close`) в ZSET сразу — таймеры переживают рестарт, а не только один из них. `OpenLot`/`CloseLot`/`CancelLot` — под `lockForUpdate`(Postgres) внутри `UnitOfWork.transaction`, bump `version`, запись в outbox в той же TX; Redis-статус (`CasService.setStatus`/`clear`) пишется СТРОГО ПОСЛЕ коммита — Postgres источник истины, Redis-кандидат (§6). `CloseLot` дополнительно оборачивается в `LockService.withLock` (Redis distributed lock) — конкурентные вызовы: проигравший тихо no-op, не ошибка; при закрытии публикует ДВА outbox-события за одну транзакцию — `lot.closing` (вход в фазу закрытия) и `lot.closed` (триггер будущей saga сеттлмента, §7, единственный routing key, забинженный на `settlement.q`). Анти-снайп: `newCloseAt = lastBidAt + antiSnipeWindowSec*1000` (гарантирует полное окно ПОСЛЕ фактической последней ставки, не от текущего `closeAt`/`now`), применяется только если реально позже текущего `closeAt`; при срабатывании — лот остаётся `open`, ZSET close-таймер переставляется тем же payload (`lotId` — стабильный идентификатор, продлевает score вместо дублирования). `status==='scheduled'` внутри `CloseLot` — намеренно бросает исключение (гонка close-before-open между тиками шедулера) — пробрасывается наружу через `AuctionSchedulerDispatcher` до `ZSetScheduler.tick()`, которая делает `ZADD GT`-реквизит — самоисцеление на следующем тике. `CancelLot` добавляет новый routing key `lot.cancelled` (нет в §8.1, биндинг очереди сознательно не добавлен — событие уходит в `auction.events`, пока без слушателя, будущие M2-07/notification добавят биндинг). API: `POST /lots`/`POST /lots/:id/cancel` (`@Roles('shipper')`), `GET /lots/:id` (любая аутентифицированная роль) — ответы маппятся через `LotResponseDto` (не отдают доменный `version`).

Архитектурная развилка: `platform/scheduler`'s `SchedulerTicker` инжектит `SCHEDULER_DISPATCH_PORT` опционально (не провайдер `SchedulerModule`), а `platform/*` не может импортировать `modules/auction` (нарушение направления зависимостей). Решение — новый `@Global()` клеевой модуль `AuctionSchedulerWiringModule` (`imports:[AuctionModule]`, регистрирует `{provide:SCHEDULER_DISPATCH_PORT, useExisting:AuctionSchedulerDispatcher}`, экспортирует токен) — по прецеденту `OUTBOX_PORT` (глобальный экспорт из одного глобального модуля виден инжектору-потребителю в другом глобальном модуле). `platform/scheduler` не тронут; направление зависимостей не нарушено — «клей» живёт на стороне домена.

По ходу реализации найден и исправлен реальный NestJS-баг (не архитектурная ошибка плана, а особенность фреймворка): `@UseGuards(JwtAuthGuard, RolesGuard)` в `LotsController` резолвит зависимости guard'ов через инжектор МОДУЛЯ-ПОТРЕБИТЕЛЯ (`AuctionModule`), а не модуля, где guard объявлен (`IdentityModule`) — `IdentityModule` экспортировал только сами классы guard'ов, но не `JwtModule`, от которого зависит `JwtAuthGuard` через `JwtService`. ## 2026-07-17 — M2-04 platform/scheduler: ZSET-планировщик отложенных команд

**Что:** Планировщик отложенного open/close лота поверх Redis ZSET (`auction:schedule:open`/`auction:schedule:close`). Отклонение от буквального скелета §9.7: вместо раздельных `zrangebyscore`→dispatch→`zrem` (три шага с race-окном — два параллельных тика увидели бы один и тот же элемент до его удаления) — `ZSetScheduler.tick()` вызывает Lua-скрипт `SCHEDULER_CLAIM_DUE`, атомарно совмещающий `ZRANGEBYSCORE`+`ZREM` в одной операции; `dispatch()` вызывается уже после атомарного claim, вне Lua. `schedule(setKey, dueAtMs, payload)` — обычный `ZADD`; `payload` обязан быть стабильным идентификатором (лотId) — повторный `schedule` того же payload обновляет score существующего члена (продление/анти-снайп), а не создаёт дубликат — это задокументировано как инвариант в JSDoc.
