# M2-03 — platform/outbox: разбор по косточкам

> Учебный конспект по задаче M2-03 (транзакционный outbox: таблица + relay-публикатор).
> Цель — чтобы к коду можно было вернуться в любой момент и освежить суть.
> Идёт от фундамента (какую именно беду он лечит) к каждому компоненту.
> Стоит на плечах двух предыдущих: [m2-01-redis-explained.md](m2-01-redis-explained.md) (Redis хранит *состояние*),
> [m2-02-messaging-explained.md](m2-02-messaging-explained.md) (RabbitMQ передаёт *события*).
> Здесь — **клей между Postgres и RabbitMQ**: как записать в БД и отправить в брокер, ничего не потеряв и не выдумав.

## 0. Четыре факта, без которых outbox — просто «ещё одна таблица»

1. **Проблема называется dual-write, и она фундаментальна.** Обработчик команды почти всегда
   хочет сделать *две* вещи разом: поменять данные в Postgres (лот перешёл в `closing`) и
   сказать миру об этом через RabbitMQ (`lot.closed`). Это две разные системы с двумя разными
   «коммитами», и **нет способа закоммитить их атомарно**. Любой порядок ломается на падении:

   ```
   Вариант А: сначала БД, потом брокер          Вариант Б: сначала брокер, потом БД
   ───────────────────────────────────          ───────────────────────────────────
   COMMIT в Postgres            ✅               publish в RabbitMQ           ✅
   ← ПРОЦЕСС УПАЛ ЗДЕСЬ ✗                         ← ПРОЦЕСС УПАЛ ЗДЕСЬ ✗
   publish в RabbitMQ    (не случился)           COMMIT в Postgres      (не случился)
   ───────────────────────────────────          ───────────────────────────────────
   Итог: данные есть, события НЕТ.               Итог: событие есть, данных НЕТ.
   Потребители не узнали → лот «завис».          Все среагировали на то, чего не было.
   ```

   Первое — **потерянное событие** (silent data loss, самое коварное — всё «зелёное», а система
   рассинхронизирована). Второе — **фантомное событие** (phantom write). Оба — про то, что
   *две записи в две системы не бывают одной транзакцией*.

2. **Ключевой трюк: превратить две записи в одну.** Раз атомарно писать в две системы нельзя —
   не будем. Пишем **обе записи в одну систему — в Postgres**. Доменная мутация и «намерение
   отправить событие» (строка в таблице `outbox`) идут в **одной транзакции БД**. Один `COMMIT` —
   либо есть и то, и другое, либо ничего. Дуальной записи больше нет структурно, а не «потому
   что мы аккуратно проверяем».

   ```
   ┌─ одна транзакция Postgres ─────────────────┐
   │  UPDATE lots SET status='closing' ...       │   ← доменная мутация
   │  INSERT INTO outbox (routing_key, payload)  │   ← «надо будет отправить lot.closed»
   └─────────────── COMMIT ─────────────────────┘   ← атомарно: оба или ни одного
                        │
                        ▼  (потом, отдельно)
              relay читает outbox → publish в RabbitMQ → помечает строку отправленной
   ```

3. **Отправка отвязана во времени — и это осознанно.** Собственно в брокер публикует **не**
   обработчик команды, а отдельный фоновый процесс — **relay** (реле). Он периодически
   заглядывает в таблицу: «есть неотправленные строки? отправь и пометь». Значит между
   `COMMIT` и реальной публикацией проходит какой-то момент — событие какое-то время лежит в БД
   «готовым к отправке». Это цена надёжности: чуть-чуть задержки в обмен на гарантию, что
   ничего не потеряется.

4. **Гарантия — «хотя бы раз» (at-least-once), а не «ровно раз».** Раз relay сначала *публикует*,
   а потом *помечает* строку — между этими двумя шагами он может упасть. Тогда строка останется
   непомеченной, и на следующем тике relay опубликует её **снова**. Дубль — это допустимо и
   ожидаемо. Гасит его **дедуп на стороне потребителя** (по `messageId`, реализация — M3-01).
   Запомни связку: **outbox гарантирует «не потеряем» (≥1 раз), дедуп гарантирует «не удвоим»
   (≤1 раз). Вместе → эффективно «ровно раз».**

**Почему пишем руками.** Как и с messaging: задача (§5 ТЗ) — *показать* паттерн. Готовые
outbox-библиотеки спрятали бы poll-loop, `SKIP LOCKED` и at-least-once за магией. Здесь всё
явно и на ладони — это и есть предмет демонстрации.

## 1. Таблица `outbox` (`outbox.entity.ts` + миграция)

Сердце паттерна — одна таблица. Каждая строка = одно событие, ждущее отправки.

```ts
@Entity('outbox')
class OutboxEntity {
  id: string          // uuid — станет messageId в RabbitMQ (это и есть ключ дедупа!)
  routingKey: string  // 'lot.closed' — куда/кому (routing key для topic-exchange)
  payload: unknown    // jsonb — тело события
  createdAt: Date     // когда положили (по нему сортируем — FIFO)
  publishedAt: Date|null   // null = ещё не отправлено; дата = отправлено. Весь статус — в этом поле
  attempts: number    // счётчик неудачных попыток публикации
}
```

Три поля несут всю логику, стоит проговорить каждое:

- **`id` = будущий `messageId`.** Не «просто первичный ключ». Тот же uuid уедет в RabbitMQ как
  `messageId`, и по нему потребитель поймёт «это я уже видел». Одна и та же строка, опубликованная
  дважды, приедет с **одинаковым** `messageId` → дедуп сработает. Идентичность события задаётся
  здесь, при вставке.
- **`publishedAt` — это и есть машина состояний строки, всего из двух состояний.** `NULL` =
  «кандидат на отправку», дата = «отправлено, забудь». Никакого отдельного `status`-поля не нужно.
- **`attempts`** — сколько раз публикация этой строки падала. Диагностика + задел на будущую
  парковку «отравленных» строк.

**Хитрость производительности — частичный индекс.** В миграции:

```sql
CREATE INDEX ... ON outbox (created_at) WHERE published_at IS NULL;
```

Relay всегда спрашивает «дай неотправленные, по возрасту». Со временем в таблице копятся
**миллионы** уже отправленных строк — а нужны только те, где `published_at IS NULL` (обычно
единицы). Частичный индекс (`WHERE published_at IS NULL`) физически содержит **только
неотправленные** строки. Выборка остаётся дешёвой независимо от того, сколько истории накопилось.
Отправил строку → она *выпадает* из индекса. Элегантно.

## 2. Пишущая сторона: `OutboxService.add` (`outbox.service.ts`)

Самый маленький файл в модуле — и в этом вся суть:

```ts
async add(manager: EntityManager, eventType: string, payload: unknown) {
  await manager.insert(OutboxEntity, {
    id: randomUUID(), routingKey: eventType, payload,
  });
}
```

Единственное, что тут важно понять — **`manager` приходит снаружи**. Это не «свежее»
подключение к БД, а `EntityManager` **чужой, уже открытой транзакции** — той самой, в которой
домен прямо сейчас меняет лот. Поэтому `INSERT` в outbox попадает в **тот же `COMMIT`**, что и
доменная мутация. Вот здесь — буквально в одной строке `manager.insert` — и происходит «слияние
двух записей в одну» из §0.2.

`add` **не создаёт** транзакцию, **не коммитит**, **не публикует** ничего. Только вставляет
строку в переданный контекст. Всё остальное — забота вызывающего (транзакция) и relay (публикация).

## 3. Как домен вызывает `add`, не зная про outbox (порт из M1-03)

Прямых `OutboxService.add(...)` в бизнес-коде ты не найдёшь. Между ними — **порт** (тот же приём,
что `DedupPort` в M2-02). Цепочка:

```
доменный handler
   │  uow.transaction(async (tx) => { ... tx.outbox.add('lot.closed', payload) ... })
   ▼
UnitOfWork.transaction()      открывает транзакцию Postgres, оборачивает manager в TransactionContext
   ▼
TransactionContext.outbox     это OutboxPort (интерфейс), а не конкретный OutboxService
   ▼
DI-токен OUTBOX_PORT   ──►  реально указывает на OutboxService (проводка в OutboxModule)
```

Смысл развязки — **домен зависит от интерфейса `OutboxPort`, а не от файла `outbox.service.ts`.**
Ключевой момент из M1-03: `UnitOfWork` инжектит порт как **`@Optional()`**, и если модуль
не подключён — подставляется `NullOutboxPort`, который **бросает понятную ошибку**:

```ts
class NullOutboxPort {
  add() { throw new Error('Outbox is not configured yet — ... Provide OUTBOX_PORT once it does.'); }
}
```

То есть с M1-03 контракт `add(manager, eventType, payload)` уже существовал — просто «звонил в
пустоту». Задача M2-03 = **заменить заглушку на реальную запись**, не тронув ни строчки в домене.
Это и есть польза от порта: шов проложили заранее, реализацию вставили потом.

## 4. Читающая сторона: `OutboxRelay.tick()` (`outbox.relay.ts`) — самое сердце

Relay — фоновый воркер. `setInterval` в `onModuleInit` дёргает `tick()` каждые
`pollIntervalMs`; в `onModuleDestroy` интервал гасится (иначе — утечка таймера при рестарте).

Один `tick()` — это **одна транзакция Postgres на весь батч**:

```ts
async tick() {
  if (this.ticking) return;          // (A) реентерабельность в ОДНОМ процессе
  this.ticking = true;
  try {
    await this.dataSource.transaction(async (manager) => {   // (B) одна TX на батч
      const rows = await repo.fetchUnpublished(manager, batchSize);   // SELECT ... FOR UPDATE SKIP LOCKED
      for (const row of rows) {
        try {
          await publisher.publish(Exchanges.events, row.routingKey, row.payload,
                                   { messageId: row.id });            // (C) публикация
          await repo.markPublished(manager, row.id);                  // (D) пометить отправленной
        } catch (err) {
          await repo.recordFailure(manager, row.id);                  // (E) attempts++, батч НЕ рушим
        }
      }
    });
  } finally { this.ticking = false; }
}
```

Разберём четыре неочевидных решения — каждое чинит конкретную гонку.

### (A) Флаг `ticking` — защита от наложения тиков в одном процессе
`setInterval` не ждёт, пока предыдущий `tick()` закончится. Если публикация батча идёт дольше
`pollIntervalMs`, таймер выстрелит снова поверх работающего тика. Флаг `ticking` заставляет
второй вызов сразу выйти. **Важно:** это защита **только внутри одного процесса** — против
нескольких инстансов приложения он бесполезен (у каждого свой флаг в своей памяти). За
многопроцессность отвечает (B)+`SKIP LOCKED`.

### (B)+(C)+(D) Почему fetch, publish и mark — в ОДНОЙ транзакции
Соблазн: вычитать строки, закоммитить, *потом* спокойно публиковать. **Так делать нельзя** — и в
коде на этот счёт есть отдельный комментарий. Причина — блокировка. `fetchUnpublished` берёт
строки через `FOR UPDATE SKIP LOCKED` (см. §5): пока транзакция жива, эти строки **залочены** от
других тиков/инстансов. Если закоммитить сразу после `SELECT`:

```
Инстанс #1: SELECT rows [1,2,3], COMMIT (лок снят!) ──► publish 1... publish 2...
Инстанс #2:              SELECT ── видит [1,2,3] РАЗБЛОКИРОВАННЫМИ ──► publish 1, 2, 3 тоже!
                         Результат: КАЖДОЕ событие ушло дважды (лишний дубль на ровном месте)
```

Держа лок до конца `publish`+`markPublished`, мы гарантируем: пока #1 возится с этими строками,
#2 их **не видит** (`SKIP LOCKED` их проскочит). Дубли из-at-least-once возможны только при
реальном падении, а не при штатной конкуренции. Отсюда правило: **лок держим ровно до момента,
когда строка помечена отправленной.**

### (E) Построчный try/catch — один сбой не рушит весь батч
Публикация строки может упасть (брокер моргнул). Если бы исключение вылетело наружу — откатилась
бы **вся** batch-транзакция, включая уже успешные `markPublished` соседних строк, и они
переопубликовались бы зря. Поэтому ошибка каждой строки ловится точечно: `recordFailure`
(`attempts++`), а цикл идёт дальше. Успешные строки в этом же батче спокойно коммитятся.

**А что если упало НЕ в publish, а сама batch-транзакция (например, в `markPublished`)?** Тогда
откатывается весь батч — строки остаются `published_at IS NULL`. На следующем тике они
переотправятся. Consumer увидит дубль (тот же `messageId`) — и это **задокументированное,
корректное** at-least-once поведение, а не баг.

## 5. Гвоздь программы: `FOR UPDATE SKIP LOCKED` (`outbox.repository.ts`)

Именно эта конструкция позволяет **запускать несколько инстансов API параллельно**, и все они
дружно разгребают одну таблицу без дублей и без общего оркестратора.

```ts
fetchUnpublished(manager, limit) {
  return manager.createQueryBuilder(OutboxEntity, 'outbox')
    .where('outbox.publishedAt IS NULL')   // только кандидаты
    .orderBy('outbox.createdAt', 'ASC')    // FIFO — старые первыми
    .limit(limit)                          // батч (batchSize)
    .setLock('pessimistic_write')          // ← FOR UPDATE: залочить выбранные строки
    .setOnLocked('skip_locked')            // ← SKIP LOCKED: чужое залоченное — пропустить
    .getMany();
}
```

Что делает связка построчно:
- **`FOR UPDATE`** (`pessimistic_write`) — «беру эти строки под свою транзакцию, никто другой их
  не тронет, пока я не закоммичу».
- **`SKIP LOCKED`** — «а если строка уже кем-то залочена — не жди её, просто **перешагни** и
  бери следующую свободную».

Вместе это раздаёт работу без спора. Наглядно — три инстанса на 6 строках:

```
outbox: [1][2][3][4][5][6]  (все published_at IS NULL)

Инстанс #1: SELECT ... LIMIT 2 FOR UPDATE SKIP LOCKED → лочит [1][2], публикует их
Инстанс #2: SELECT ... LIMIT 2 → [1][2] залочены, ПРОПУСКАЕТ → лочит [3][4]
Инстанс #3: SELECT ... LIMIT 2 → [1..4] залочены, ПРОПУСКАЕТ → лочит [5][6]

Итог: 6 строк, 6 публикаций, 0 дублей, 0 простоя. Никто никого не ждал.
```

Без `SKIP LOCKED` инстансы #2 и #3 **блокировались** бы на строке [1], ожидая коммита #1 —
конкуренция превратилась бы в очередь. С ним — честный параллелизм. Это классический паттерн
«очередь задач поверх обычной SQL-таблицы».

- **`markPublished`** — `UPDATE outbox SET published_at = now() WHERE id = ...`. Строка выпадает
  из частичного индекса, `tick()` её больше не увидит.
- **`recordFailure`** — `attempts = attempts + 1`. Через `manager.increment` (атомарный `+1` на
  стороне БД, без гонки read-modify-write).

## 6. Полная прогонка на конкретном сценарии

Соберём всё вместе. Домен закрывает лот `L-42`, `pollIntervalMs = 500`.

**T0 — обработчик команды (одна транзакция):**
```
BEGIN
  UPDATE lots SET status='closing', version=8 WHERE id='L-42' AND version=7
  INSERT INTO outbox (id='e-99', routing_key='lot.closed', payload={lotId:L-42}, published_at=NULL)
COMMIT                       ← атомарно. Событие «e-99» теперь надёжно лежит в БД.
```
Даже если сервис упадёт сразу после `COMMIT` — событие **не потеряно**, оно в таблице.

**T0+300ms — relay тикает:**
```
BEGIN
  SELECT ... WHERE published_at IS NULL ... FOR UPDATE SKIP LOCKED  → [e-99]  (лочим)
  publisher.publish('auction.events', 'lot.closed', {lotId:L-42}, {messageId:'e-99'})   ✅
  UPDATE outbox SET published_at=now() WHERE id='e-99'
COMMIT                       ← строка помечена, выпала из индекса
```
`notification.q` / `listing.q` получают `lot.closed` с `messageId='e-99'`. Готово, ровно один раз.

**Ветка сбоя — упали между publish и markPublished:**
```
BEGIN
  SELECT → [e-99] (лочим)
  publisher.publish(...)     ✅  ← брокер сообщение УЖЕ принял
  ← ПРОЦЕСС УПАЛ ✗           markPublished не выполнился, COMMIT не случился
ROLLBACK (неявный)           ← published_at остался NULL, лок снят
```
Через 500ms следующий тик снова видит `e-99` (всё ещё `NULL`) → публикует **повторно** с тем же
`messageId='e-99'`. Consumer по дедупу (M3-01) видит «e-99 уже обрабатывал» → **глушит дубль**.
Итог для мира: событие доставлено ровно один раз, хотя опубликовано дважды. Вот она,
связка «outbox (≥1) + дедуп (≤1) = ровно раз» в действии.

## 7. Конфиг

Две `OUTBOX_*` env → секция `AppConfigService.outbox`:

| Поле | env | Что | Роль |
|---|---|---|---|
| `pollIntervalMs` | `OUTBOX_POLL_INTERVAL_MS` | как часто тикать relay | латентность vs нагрузка на БД |
| `batchSize` | `OUTBOX_BATCH_SIZE` | сколько строк за тик | throughput vs длина транзакции |

**Урок из ревью (записан в worklog как грабли).** На первом проходе у обоих параметров была
только *нижняя* граница. Security-review отметил WARNING: без *верхней* границы опечатка в
конфиге (`OUTBOX_BATCH_SIZE=1000000` или `POLL_INTERVAL_MS=0`) даёт гигантский батч / busy-loop.
Исправлено: `.max(1000)` на батч, `.max(60_000)` + `.min(50)` на интервал. Отсюда правило,
которое перекочевало в M2-04: **у числового env-параметра всегда обе границы — и `.min`, и `.max`.**

## 8. Что осознанно отложено (честные заглушки и грабли)

- **Нет потолка на `attempts`.** «Отравленная» строка (payload, который брокер всегда отвергает)
  будет ретраиться **вечно** на каждом тике. Признано не блокирующим: `Publisher.publish` не
  делает внутренних ретраев, так что на брокер это лишней нагрузки не льёт. Кандидат на будущее —
  парковка в dead-letter после N попыток (для того `attempts` и копится).
- **`payload: jsonb` без ограничения размера.** Ответственность вызывающего домена, не этого
  слоя — outbox не знает и не должен знать, что за событие в него кладут.
- **Дедуп — не здесь.** Outbox отвечает только за «≥1 раз». «≤1 раз» — это `DedupPort` на
  потребителе, живая Redis-реализация — **M3-01**. До неё дубли из at-least-once реально доедут
  до consumer'а (в тестах это специально проверяется как ожидаемое поведение).
- **`setInterval`, а не `@nestjs/schedule`.** Тянуть зависимость ради одного цикла — незачем.
  Тот же приём, что в `SchedulerTicker` (M2-04, который прямо скопировал этот паттерн реентерабельности).

## Карта в голове

| Компонент | Задача | Механика |
|---|---|---|
| `outbox` (таблица) | хранить «намерения отправить» | `published_at IS NULL` = кандидат; частичный индекс держит выборку дешёвой |
| `OutboxService.add` (§2) | записать событие в чужую TX | `manager.insert` в переданный `EntityManager` → один COMMIT с доменом |
| `OUTBOX_PORT` (§3) | развязать домен от реализации | порт из M1-03; `NullOutboxPort` бросал ошибку до M2-03 |
| `OutboxRelay.tick` (§4) | опубликовать и пометить | одна TX на батч: fetch → publish → markPublished; `ticking`-флаг; построчный try/catch |
| `fetchUnpublished` (§5) | раздать работу без дублей | `FOR UPDATE SKIP LOCKED` → параллельные инстансы не спорят и не ждут |

**Три идеи, которые держат весь M2-03:**
1. **Две записи → одна транзакция.** Dual-write нерешаем атомарно, поэтому пишем и данные, и
   событие в *один* Postgres-COMMIT. Потерянных/фантомных событий нет **структурно**.
2. **At-least-once + дедуп = ровно раз.** Relay честно допускает дубли (publish до mark, падение
   между ними). «Не удвоить» — отдельная забота потребителя по `messageId`. Разделение
   ответственности, а не один хитрый механизм.
3. **`FOR UPDATE SKIP LOCKED` = очередь задач на голом SQL.** Один приём даёт горизонтальное
   масштабирование relay без оркестратора: инстансы разбирают строки, залоченное — перешагивают.
