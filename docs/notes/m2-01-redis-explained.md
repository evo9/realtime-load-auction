# M2-01 — platform/redis: разбор по косточкам

> Учебный конспект по задаче M2-01 (Lock, RateLimiter, PubSub, CAS-каркас).
> Цель — чтобы к коду можно было вернуться в любой момент и освежить суть.
> Идёт от фундамента (Redis + Lua с нуля) к каждому сервису.

---

## 0. Три факта про Redis, без которых остальное — магия

1. **Redis — это один большой словарь «ключ → значение» в оперативной памяти.**
   Значение бывает не только строкой:
   - строка: `lot:42:status` → `"open"`
   - хеш (словарь внутри значения): `lot:42:high` → `{amount, carrierId, bidId}`
   - sorted set / ZSET — множество, где у каждого элемента есть числовой **score** (вес),
     и оно всегда отсортировано по этому весу. На нём построен rate-limiter.

2. **Redis однопоточный.** Команды выполняются строго по одной, в очередь. Пока идёт одна
   команда, никакая другая не влезет в середину. Отсюда растёт вся идея «атомарности».

3. **Зачем Redis, а не Postgres, для ставок.** Горячий путь (кто сейчас лидирует) должен
   отвечать за микросекунды. Postgres — источник истины (медленный, надёжный),
   Redis — быстрый кандидат. `lot:{id}:high` в Redis — *кандидат*, а не истина (§6 ТЗ).

## 1. Зачем тут Lua и что такое атомарность (разблокирует всё)

Логика вида «если ставка лучше — запиши её» в обычном коде это два шага:

```
1. прочитать текущую лучшую цену   (GET)
2. если новая меньше — записать     (SET)
```

Проблема: Redis однопоточный, но приложение — нет. Между шагом 1 и 2 другой процесс может
вклиниться со своей командой → **гонка (race condition)**: ты прочитал «500», решил «моя 490
лучше», но пока думал, кто-то записал 480, и ты своим `SET 490` затираешь чужие 480.

Решение: **склеить оба шага в одну команду, которую Redis выполнит целиком, никого не впуская
в середину.** Такую склейку пишут на Lua — языке, встроенном внутрь Redis. «Неделимо, от
начала до конца, никто не вклинится» = **атомарность**. Ради неё и тащим Lua.

### Синтаксис Lua — ровно 6 конструкций, которые надо узнавать

```lua
local x = 5                       -- объявить локальную переменную
redis.call('GET', KEYS[1])        -- вызвать команду Redis изнутри скрипта
KEYS[1]                           -- 1-й КЛЮЧ, переданный снаружи (нумерация с 1!)
ARGV[1]                           -- 1-й АРГУМЕНТ (не-ключ), переданный снаружи
tonumber(ARGV[1])                 -- строку "5" превратить в число 5
if count < limit then ... end     -- условие; блок закрывается словом end
return {1, 'accepted'}            -- вернуть массив (в Lua он зовётся "таблица")
```

Три вещи, которые всех путают:

1. **Нумерация с единицы.** `KEYS[1]`, `ARGV[1]` — первые. Не `[0]`.
2. **KEYS и ARGV — разные корзины.** Всё, что снаружи объявлено ключом, идёт в `KEYS`,
   остальное — в `ARGV`. Число ключей объявляется явно (`numberOfKeys` / первый аргумент
   `eval`). Redis должен знать, какие именно ключи трогает скрипт.
3. **Из Lua всё снаружи приходит строками.** `ARGV[1]` это `"500"`, не число → перед
   сравнением `tonumber(...)`. И наоборот, в JS числа заворачивают: `String(amount)`.

Возврат: Lua-таблица `{1, 'accepted'}` на стороне JS станет массивом `[1, 'accepted']`.
Булевых в этом канале нет, поэтому договорились: **1 = да, 0 = нет** → в коде `accepted === 1`.

## 2. Мелочи-связки: token, keys, module

**`redis-client.token.ts`**
```ts
export const REDIS_CLIENT = Symbol('REDIS_CLIENT');
```
«Уникальный штрих-код» на коробке с Redis-подключением. NestJS раздаёт зависимости по имени;
`Symbol` уникален физически, поэтому `@Inject(REDIS_CLIENT)` гарантированно получит именно
твой клиент, ничего не перепутается.

**`redis-keys.ts`** — единый справочник имён ключей.
```ts
lotHigh:   (lotId) => `lot:${lotId}:high`,
lotStatus: (lotId) => `lot:${lotId}:status`,
lotLock:   (lotId) => `lot:${lotId}:lock`,
rateLimit: (carrierId, lotId) => `ratelimit:${carrierId}:${lotId}`,
```
Имя ключа записано **в одном месте** → опечатка вроде `lot-42-high` невозможна, два куска кода
не разъедутся по разным ячейкам. Имена зафиксированы в §8 ТЗ.

**`redis.module.ts`** — два важных момента:
- одно подключение к Redis создаётся один раз (`useFactory`) под биркой `REDIS_CLIENT`,
  и все 4 сервиса получают **его же** через `@Inject`. Одно соединение на всех — дёшево и верно.
- `RedisLifecycle.onModuleDestroy` → `client.quit()` — аккуратно закрываем при остановке.
- `@Global()` — модуль виден всему приложению без повторного импорта.

## 3. LockService — распределённый замок

**Задача:** гарантировать, что действие делает только один процесс за раз
(ТЗ: «лот закрывается/сеттлится ровно один раз»).

```ts
async acquire(key, ttlMs): Promise<Lock | null> {
  const token = randomUUID();
  const result = await this.client.set(key, token, 'PX', ttlMs, 'NX');
  return result === 'OK' ? { key, token } : null;
}
```

Разбор `set(key, token, 'PX', ttlMs, 'NX')` — одна команда `SET` с двумя флагами:

- **`NX`** = «Not eXists» — запиши, **только если ключа ещё нет**. Это и есть весь замок.
  Первый успел → `'OK'`. Второй опоздал → `null`. Атомарно, т.к. Redis однопоточный.
  → покрывает DoD «два конкурентных захвата — только один успешен».
- **`PX ttlMs`** = TTL в **миллисекундах**. `EX` — секунды, `PX` — миллисекунды, разница
  только в единицах. Замок короткоживущий (десятки-сотни мс), секунды слишком грубо.
- **Зачем TTL:** страховка от смерти держателя. Взял замок и упал — без TTL ключ висел бы
  вечно (deadlock). С TTL Redis сам снимет замок. TTL = «замок с автоспуском».

**Зачем `token = randomUUID()`** — ради безопасного релиза:

```ts
async release(lock): Promise<boolean> {
  const result = await this.client.eval(LOCK_RELEASE, 1, lock.key, lock.token);
  return result === 1;
}
```
```lua
-- LOCK_RELEASE
if redis.call('GET', KEYS[1]) == ARGV[1] then
  return redis.call('DEL', KEYS[1])
else
  return 0
end
```

Опасный сценарий без токена:
1. Процесс **A** взял замок, TTL 200мс.
2. A затормозил на 250мс → TTL истёк, Redis снял замок.
3. Процесс **B** увидел свободно, взял замок себе.
4. A очнулся, делает `DEL key` → **удаляет замок B**, хотя B ещё работает. Двое держат замок.

Защита: каждый кладёт в значение **свой токен**. Релиз проверяет «там мой токен? → удаляю,
иначе не трогаю (`return 0`)». Снова два шага («прочитать» + «удалить если совпал»), между
которыми возможна гонка → **обязан быть Lua** (атомарная «проверь-и-удали»).

`eval(LOCK_RELEASE, 1, lock.key, lock.token)`: `1` = «дальше 1 ключ» → `lock.key`=`KEYS[1]`,
`lock.token`=`ARGV[1]`.

```ts
async withLock(key, ttlMs, fn) {
  const lock = await this.acquire(key, ttlMs);
  if (!lock) return null;                 // не взял — не делаю ничего
  try { return await fn(); }
  finally { await this.release(lock); }   // ЧТО БЫ НИ СЛУЧИЛОСЬ — отпускаю
}
```
`finally` критичен: даже если `fn` бросит ошибку, замок вернётся. `null` = «замок занят».

## 4. CasService — и честно про то, что он пока каркас

**CAS = Compare-And-Set** — «прими новую лучшую ставку, только если она реально лучше текущей,
и всё атомарно». В reverse-аукционе «лучше» = «меньше».

**Текущая версия скрипта (M2-01) — заглушка:**
```lua
-- CAS_BEAT_HIGH_BID (КАРКАС)
redis.call('HSET', KEYS[1], 'amount', ARGV[1], 'carrierId', ARGV[2], 'bidId', ARGV[3])
return {1, 'accepted'}
```
Скрипт **безусловно** пишет ставку в хеш `lot:{id}:high` и **всегда** возвращает «принято».
Нет ни сравнения цены, ни проверки статуса. `KEYS[2]` (статус) передан, но не используется.

**Это не баг — так в задаче.** M2-01: *«CasService — метод-каркас без бизнес-сравнения,
заглушка-accept, доделать в M3-02»*. В M3-02 внутри Lua появится настоящая логика:
```
если статус ≠ "open"              → {0, 'closed'}
если новая цена НЕ меньше текущей  → {0, 'too_low'}
иначе                             → HSET ... ; {1, 'accepted'}
```

Что реально важно понять в M2-01 — **механику загрузки скрипта** (она общая с rate-limiter):
```ts
constructor(@Inject(REDIS_CLIENT) client) {
  client.defineCommand('casBeatHighBid', { numberOfKeys: 2, lua: CAS_BEAT_HIGH_BID });
  this.commands = client as unknown as CasCommands;
}
```
- `defineCommand` регистрирует Lua-скрипт как именованную команду (детали — раздел 7).
- `numberOfKeys: 2` → первые 2 аргумента вызова это KEYS, остальные ARGV.
- `as unknown as CasCommands` — только для TypeScript: метод добавлен в рантайме, TS о нём не
  знает; строкой обещаем компилятору, что метод есть. На поведение не влияет.

```ts
async tryBeatHighBid(lotId, amount, carrierId, bidId): Promise<CasResult> {
  const [accepted, reason] = await this.commands.casBeatHighBid(
    RedisKeys.lotHigh(lotId),    // KEYS[1]
    RedisKeys.lotStatus(lotId),  // KEYS[2]
    String(amount),              // ARGV[1]  (число → строку)
    carrierId, bidId,            // ARGV[2], ARGV[3]
  );
  return { accepted: accepted === 1, reason };
}
```
Метод — просто «труба» между кодом и Lua. В M3-02 меняется скрипт, труба остаётся.

## 5. RateLimiter — ZSET и скользящее окно

**Задача:** не дать перевозчику завалить лот ставками. Правило «не больше N ставок за окно M мс».

**Инструмент — ZSET.** Кладём по одному элементу на каждую ставку, **score = время ставки в мс**.
Тогда ZSET автоматически = отсортированная по времени история ставок.

**Скользящее окно (sliding window).** Не «сброс счётчика по расписанию» (на границе можно всадить
N в конце одного окна и N в начале следующего), а всегда «последние M мс от текущего момента».
Окно едет вместе с часами: перед подсчётом выкидываем всё старше `now − window`.

```lua
-- RATE_LIMIT_HIT
-- KEYS[1]=zset  ARGV[1]=now  ARGV[2]=window  ARGV[3]=limit  ARGV[4]=member
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, now - window)  -- (а) метла: выкинуть протухшее
local count = redis.call('ZCARD', KEYS[1])                -- (б) сколько в окне сейчас
if count < limit then
  redis.call('ZADD', KEYS[1], now, ARGV[4])               -- (в) добавить: score=now, элемент=member
  redis.call('PEXPIRE', KEYS[1], window)                  --     TTL-гигиена: idle-ключ сам исчезнет
  return {1, limit - count - 1}                           --     разрешено; сколько ещё осталось
end
return {0, 0}                                             -- лимит исчерпан; запрещено
```

Команды:
- `ZREMRANGEBYSCORE key 0 (now-window)` — удалить элементы со score в `[0, now-window]`
  (всё, что уехало за левый край окна). Это «сдвиг окна».
- `ZCARD` — размер множества = число ставок в текущем окне.
- `ZADD key score member` — добавить ставку (score=время).
- `PEXPIRE key window` — TTL ключу = ширина окна (в мс). Перестал ставить — ключ сам исчезнет.
- `{1, remaining}`: было `count`, добавили 1 → свободно `limit-count-1`.

**Целиком в Lua** — снова «почисти → посчитай → реши → добавь» на одних данных; раздели на
JS-вызовы → гонка между «4 < 5» и «добавил», двое проскочат лимит.

### Пошаговая прогонка (лимит 3, окно 10 000 мс, ключ `ratelimit:c1:42`)

```
A  t=0     чистим [0,-10000]→ø   ZCARD 0  0<3 ✅ rem=2   ZSET: A(0)
B  t=2000  чистим [0,-8000] →ø   ZCARD 1  1<3 ✅ rem=1   ZSET: A(0) B(2000)
C  t=3000  чистим [0,-7000] →ø   ZCARD 2  2<3 ✅ rem=0   ZSET: A(0) B(2000) C(3000)
D  t=4000  чистим [0,-6000] →ø   ZCARD 3  3<3 ❌ {0,0}   ZSET: A(0) B(2000) C(3000)  (D НЕ добавлен)
E  t=11000 чистим [0,1000]→убрал A  ZCARD 2  2<3 ✅ rem=0  ZSET: B(2000) C(3000) E(11000)
```

- При **отказе (D)** в ZSET ничего не пишем → отклонённые попытки не копятся и не наказывают в будущем.
- В момент **E (11000)** окно = `(1000, 11000]`. A(score 0) уехал за левый край → метла его смыла →
  освободился ровно один слот. Вот оно, «скольжение»: слот освободился сам, без сброса счётчика.

```
      A          B     C              E
   ───●──────────●─────●──────────────●────►  время (мс)
      0        2000  3000           11000
   окно (last 10s):  (1000 ────────────── 11000]   ← A(0) левее края → выметается
```

Три опоры: **score=время** (история сама сортируется), **ZREMRANGEBYSCORE** (метла, двигающая
левый край), **ZCARD после метлы** («сколько в окне сейчас») → `< limit`.

**Зачем `member = ${now}-${uuid}`:** две ставки в одну мс имеют одинаковый score; ZSET —
множество, различает элементы по **member**, не по score. Без UUID второй элемент с тем же
member перезаписал бы первый, ставка потерялась бы из подсчёта. UUID → каждая ставка отдельна.

### КЛЮЧ = контейнер (ключевая мысль)

Ключ `ratelimit:c1:42` — это **и есть** ZSET. Не «общий ZSET, а внутри ставки с пометкой c1:42».
Сам ключ = отдельный ZSET, в нём **только** ставки `c1` по лоту `42`. Весь скрипт бьёт по `KEYS[1]`.

```
ratelimit:c1:42 → ZSET {ставки c1 по лоту 42}   ← своё окно, свой лимит
ratelimit:c1:99 → ZSET {ставки c1 по лоту 99}   ← другое окно, независимое
ratelimit:c2:42 → ZSET {ставки c2 по лоту 42}   ← третье, тоже своё
```

Следствия:
- `c1`, упёршийся в лимит на лоте 42, спокойно ставит на лоте 99 (другой ключ). Лимиты не перетекают.
- `c1` и `c2` на одном лоте 42 не мешают друг другу (разные ключи). Лимит персональный.
- **Гранулярность лимита = гранулярность ключа.** `{carrier}` → на перевозчика вообще;
  `{lot}` → на лот суммарно; здесь `{carrier}:{lot}` → «этот перевозчик на этом лоте».

Итого: ключ отвечает «в каком окне считаем», member — «какая именно ставка внутри окна».

## 6. PubSub — «радио» внутри Redis (Lua нет)

Pub/Sub = издатель вещает в **канал** (строку-имя), все **подписчики** канала получают сообщение.
В проекте: одна новая лучшая ставка → мгновенно разослать всем, кто смотрит лот (realtime fan-out).

```ts
constructor(@Inject(REDIS_CLIENT) private publisher: Redis) {
  this.subscriber = publisher.duplicate();
  this.subscriber.on('message', (channel, message) => this.dispatch(channel, message));
}
```

**Почему ДВА соединения** — главная неочевидность. Правило Redis: **как только соединение
подписалось на канал, оно уходит в «режим прослушки» и больше не может выполнять обычные
команды** (`GET`/`SET`/`publish`). Поэтому нужны двое: `publisher` работает и вещает,
`subscriber = publisher.duplicate()` только слушает. Это и есть «отдельные коннекты на pub/sub».

```ts
private handlers = new Map<string, Set<PubSubHandler>>();  // канал → набор обработчиков

publish(channel, payload) {
  return this.publisher.publish(channel, JSON.stringify(payload));  // Redis гоняет строки → stringify
}

async subscribe(channel, handler) {
  const set = this.handlers.get(channel);
  if (set) set.add(handler);                              // уже подписаны — просто +обработчик
  else {
    this.handlers.set(channel, new Set([handler]));
    await this.subscriber.subscribe(channel);             // ПЕРВЫЙ подписчик → реальный SUBSCRIBE
  }
  return async () => { await this.unsubscribe(channel, handler); };  // возвращаем «рубильник»
}

private async unsubscribe(channel, handler) {
  const set = this.handlers.get(channel);
  if (!set) return;
  set.delete(handler);
  if (set.size === 0) {                                   // ушёл ПОСЛЕДНИЙ → реальный UNSUBSCRIBE
    this.handlers.delete(channel);
    await this.subscriber.unsubscribe(channel);
  }
}

private dispatch(channel, message) {
  const set = this.handlers.get(channel);
  if (!set) return;
  let payload;
  try { payload = JSON.parse(message); }
  catch (e) { this.logger.error(...); return; }           // битый JSON не роняет сервис
  for (const handler of set) {
    try { handler(payload); }
    catch (e) { this.logger.error(...); }                 // упавший обработчик не мешает остальным
  }
}

async onModuleDestroy() { await this.subscriber.quit(); }  // чисто убираем за собой
```

Тонкости:
- Реальный `SUBSCRIBE`/`UNSUBSCRIBE` в Redis — только на **первого/последнего** слушателя канала.
  Внутри одного процесса несколько обработчиков делят одну подписку. Экономно.
- `subscribe` возвращает **функцию отписки** («рубильник»): подписался — получил рубильник,
  дёрнул, когда надоело (юзер закрыл вкладку лота).
- Два `try/catch` в `dispatch` — устойчивость realtime: один кривой клиент не роняет рассылку всем.

## 7. eval vs defineCommand (evalsha) — почему по-разному

- **`client.eval(SCRIPT, numKeys, ...)`** шлёт в Redis **весь текст скрипта при каждом вызове**.
- **`client.defineCommand('name', { lua })`** считает **SHA-хеш** скрипта; при вызове шлёт
  `EVALSHA <хеш> ...` — **только хеш**, не весь текст. Redis держит скрипты в кэше по хешу.
  Если Redis перезапустился и потерял кэш → ответ `NOSCRIPT` → ioredis сам подгружает полный
  текст и повторяет. То есть «короткий хеш + само чинится» + красивый типизированный метод.

**Почему в проекте по-разному** (вопрос не корректности — оба работают):
- `casBeatHighBid`, `rateLimitHit` — **горячий путь**, дёргаются на каждую ставку (тысячи/сек).
  Гонять 40 символов хеша вместо всего скрипта — заметная экономия → `defineCommand`.
- `LOCK_RELEASE` — редко (раз на закрытие/сеттл лота) и это 5 строк → экономить не на чем →
  простой `eval` без возни с регистрацией.

Правило: **evalsha там, где горячо и часто; eval там, где редко и мелко.** Это стилевой выбор,
не необходимость — `release` можно было бы тоже перевести на `defineCommand`, ничего бы не сломалось.

## Карта в голове

| Примитив     | Задача                         | Механика                                              |
|--------------|--------------------------------|-------------------------------------------------------|
| LockService  | «делает только один»           | `SET NX` + `PX`-TTL + токен + Lua-release              |
| CasService   | «принять лучшую ставку»        | M2-01 заглушка-accept; логика сравнения — в M3-02      |
| RateLimiter  | «не больше N за окно»          | ZSET, score=время, скользящее окно via ZREMRANGEBYSCORE|
| PubSub       | «радио» / realtime fan-out     | 2 соединения (слушать и работать нельзя одновременно)  |

**Lua** даёт атомарность там, где логика = цепочка «прочитал → решил → записал» и нельзя пускать
никого в середину: lock (проверь-и-удали), CAS (сравни-и-запиши), rate-limit (почисти-посчитай-добавь).
